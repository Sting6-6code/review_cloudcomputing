## 核心概念

### 1. 云存储分类

| 类型 | 描述 | AWS | GCP | Azure |
|------|------|-----|-----|-------|
| **块存储 (Block)** | 虚拟硬盘，挂载到单个 VM | EBS | Persistent Disk | Managed Disks |
| **文件存储 (File)** | 共享网络文件系统，多 VM 挂载 | EFS | Filestore | Azure Files |
| **对象存储 (Object)** | 无限扩展，通过 HTTP API 访问 | S3 | Cloud Storage | Blob Storage |
| **关系型数据库 (Relational)** | 结构化数据，ACID 事务 | RDS | Cloud SQL | Azure SQL |
| **NoSQL** | 灵活 schema，高吞吐 | DynamoDB | Firestore | Cosmos DB |

#### S3 存储层级 (Storage Classes)

```
Standard → Standard-IA → Glacier Instant Retrieval → Glacier Flexible → Glacier Deep Archive
  (热数据)    (低频访问)      (毫秒级恢复)              (分钟级恢复)         (12小时恢复，最便宜)
```

- 所有层级均保证 **11个9** 耐久性 (99.999999999%)
- **Intelligent-Tiering**：自动在访问层和非访问层之间迁移，无需手动管理
- **生命周期策略 (Lifecycle Policy)**：自动根据时间迁移对象到更低成本层

#### CAP 定理

分布式系统只能同时满足其中两项：
- **C**onsistency（一致性）
- **A**vailability（可用性）
- **P**artition Tolerance（分区容错）

NoSQL 通常选择 AP（可用性 + 分区容错），牺牲强一致性（最终一致性）。

### 2. DNS（域名系统）

#### 层级结构（倒树）

```
Root (.) → TLD (.com, .org) → Second-Level Domain (example.com) → Subdomain (www.example.com)
```

- **根域 (Root Zone)**：由 ICANN 管理，13 个根服务器集群
- **权威 DNS (Authoritative NS)**：最终回答域名对应 IP
- **递归解析器 (Recursive Resolver)**：代替客户端查询，缓存结果

#### 解析流程

```
客户端 → 递归解析器 → 根服务器 → TLD 服务器 → 权威 NS → 返回记录
```

- **递归查询 (Recursive)**：解析器一次性完成所有查询，返回最终答案
- **迭代查询 (Iterative)**：每步返回"去问谁"的指引，客户端自己跟进

#### 常用记录类型

| 记录类型 | 用途 |
|---------|------|
| **A** | 域名 → IPv4 地址 |
| **AAAA** | 域名 → IPv6 地址 |
| **CNAME** | 别名 → 另一个域名（不能用于 Zone Apex） |
| **MX** | 邮件服务器地址（含优先级） |
| **NS** | 授权名称服务器 |
| **TXT** | 任意文本（SPF/DKIM/DMARC 验证） |
| **SOA** | Zone 起始权威记录（序列号、刷新间隔） |
| **CAA** | 授权哪些 CA 颁发证书 |

#### TTL 策略

- **短 TTL（5分钟）**：适合变更频繁的场景（蓝绿部署、故障切换）
- **长 TTL（24小时）**：减少 DNS 查询量，降低延迟，但切换慢
- **胶水记录 (Glue Record)**：当 NS 本身在被委派的 zone 内时，需要在父区提前注册 IP，避免循环查询

#### AWS Route 53

- **公有区 (Public Hosted Zone)**：互联网解析
- **私有区 (Private Hosted Zone)**：VPC 内部解析
- **别名记录 (Alias Record)**：AWS 专有，指向 AWS 资源（ELB/S3/CloudFront），可用于 Zone Apex，无额外查询费用

### 3. CDN（内容分发网络）

#### 工作原理

```
用户请求 → 最近的边缘节点 (PoP) → 
  命中缓存 (Cache Hit) → 直接返回
  未命中 (Cache Miss) → 回源 (Origin Fetch) → 缓存 → 返回
```

- **源服务器 (Origin)**：内容的原始来源（S3, EC2, 自定义服务器）
- **边缘节点 (Edge / PoP)**：分布在全球的缓存服务器

#### 路由方式

| 方式 | 描述 |
|------|------|
| **Anycast** | 同一 IP 在全球多点广播，BGP 自动选最近节点（CDN 常用） |
| **Unicast** | 一对一路由，传统方式 |

#### 缓存控制

- **Cache-Control: max-age=3600**：边缘节点缓存 1 小时
- **Cache-Control: no-store**：禁止缓存（私密内容）
- **缓存失效 (Cache Invalidation)**：主动清除过期内容（CloudFront Invalidation API）
- **缓存破坏 (Cache Busting)**：URL 加版本号/哈希，强制用户获取新内容（如 `app.v2.js`）
- **负缓存 (Negative Caching)**：缓存 404 等错误响应，避免回源压力

#### 主流 CDN 对比

| 服务 | 特点 |
|------|------|
| **AWS CloudFront** | 400+ PoPs；两层架构（边缘 + 区域缓存）；支持 Lambda@Edge |
| **GCP Cloud CDN** | 需绑定 HTTP(S) 负载均衡；支持负缓存 |
| **Azure CDN** | 与 Azure Front Door 集成 |

### 4. SRE（站点可靠性工程）

#### 核心三角

```
SLI → SLO → SLA
(度量指标)  (内部目标)  (对外合同)
```

- **SLI (Service Level Indicator)**：实际可量化的指标（如请求成功率、延迟 P99）
- **SLO (Service Level Objective)**：内部可靠性目标（如成功率 ≥ 99.9%）
- **SLA (Service Level Agreement)**：与客户的合同承诺，违约有赔偿后果

#### 错误预算 (Error Budget)

```
Error Budget = 1 - SLO
例：SLO = 99.9% → Error Budget = 0.1% = 每月约 43.8 分钟
```

- 预算用完 → 冻结新功能发布，专注稳定性
- 预算充足 → 可以加快迭代速度

#### 可用性对照表

| 可用性 | 年度宕机时长 |
|--------|------------|
| 99% | 87.6 小时 |
| 99.9% | 8.7 小时 |
| 99.99% | 52 分钟 |
| 99.999% | 5.2 分钟 |

#### Toil（辛劳）

需消除的手动重复性工作：手动、可自动化、战术性的、与服务增长成比例的工作。

#### 事后复盘 (Postmortem)

- **无责文化 (Blameless)**：关注系统，不追究个人
- 记录：时间线、根因、影响、改进措施

### 5. 可观测性（Observability）

**三大支柱**：日志 (Logs) + 指标 (Metrics) + 追踪 (Tracing/Alerting)

#### 日志级别（从低到高）

```
ALL < TRACE < DEBUG < INFO < WARN < ERROR < FATAL < OFF
```

| 级别 | 使用场景 |
|------|---------|
| DEBUG | 开发调试，不上生产 |
| INFO | 正常业务流程（用户登录、订单创建） |
| WARN | 可恢复的异常（重试成功、配置缺失） |
| ERROR | 需要关注的失败（DB 连接失败） |
| FATAL | 不可恢复，应用崩溃 |

**日志最佳实践**：
- **禁止记录 PII**（个人身份信息：邮箱、密码、信用卡）
- 使用 **JSON 格式**，便于机器解析
- 时间戳用 **UTC**，避免时区混乱
- 包含 **correlation ID** 便于跨服务追踪

#### 指标类型

| 类型 | 描述 | 示例 |
|------|------|------|
| **Counter（计数器）** | 单调递增 | 总请求数、错误数 |
| **Gauge（仪表）** | 可增可减的当前值 | CPU 使用率、内存 |
| **Timer/Histogram** | 延迟分布 | P50/P95/P99 延迟 |

**指标后端**：Prometheus（拉取模式）、Graphite（推送模式）

#### 告警严重级别

| 级别 | 说明 |
|------|------|
| Low | 记录，无需立即响应 |
| Moderate | 工作时间内处理 |
| High | 立即响应，可能需要 on-call |

### 6. 邮件安全（Email Authentication）

#### 发送流程

```
应用 → SMTP 服务器 (SES) → 收件方 MTA → 收件箱/垃圾箱
```

SMTP 本身**无内置认证**，必须依赖 SPF/DKIM/DMARC 防伪造。

#### 三大邮件认证协议

| 协议 | 机制 | 存储位置 | 特点 |
|------|------|---------|------|
| **SPF** | IP 白名单：允许哪些 IP 发送该域邮件 | DNS TXT 记录 | 转发时会失效（Envelope From 变化）；最多 10 次 DNS 查找 |
| **DKIM** | 非对称加密签名：私钥签名邮件头，收件方用公钥验证 | 签名在邮件头；公钥在 DNS TXT | 转发后签名依然有效 |
| **DMARC** | 策略层：SPF/DKIM 对齐检查 + 策略执行 | DNS TXT 记录 | p=none（监控）/ p=quarantine（垃圾箱）/ p=reject（拒绝）|

**DMARC 对齐 (Alignment)**：From 域必须与 SPF 的 Envelope From 或 DKIM 签名域一致。

#### AWS SES 健康指标

| 指标 | 警告阈值 | 暂停阈值 |
|------|---------|---------|
| 退信率 (Bounce Rate) | >5% | >10% |
| 投诉率 (Complaint Rate) | >0.1% | >0.5% |

**SES Sandbox**：默认只能发到验证过的邮箱；需申请 Production Access 才能发给任意用户。

**垃圾邮件陷阱 (Spam Traps)**：已废弃的邮箱被 ISP 回收，发到这些地址会严重损害发件人声誉。

### 7. 负载均衡与弹性伸缩（Scaling）

#### 负载均衡层级

| 层级 | 处理内容 | 典型服务 |
|------|---------|---------|
| **L4（传输层）** | TCP/UDP，仅看 IP + 端口 | AWS NLB、GCP TCP LB |
| **L7（应用层）** | HTTP，可看 URL、Header、Cookie | AWS ALB、GCP HTTP(S) LB、Azure App Gateway |

#### 主流负载均衡服务对比

| 服务 | 厂商 | 层级 | 特点 |
|------|------|------|------|
| **ALB** (Application LB) | AWS | L7 | 区域级；基于路径/主机名路由；支持 WebSocket |
| **NLB** (Network LB) | AWS | L4 | 超高性能（百万 RPS）；静态 IP；支持 TLS pass-through |
| **GWLB** (Gateway LB) | AWS | L3 | 用于防火墙/IDS 等第三方虚拟设备 |
| **HTTP(S) LB** | GCP | L7 | **全球 Anycast**；单 VIP；集成 Cloud CDN |
| **App Gateway** | Azure | L7 | 区域级；内置 WAF |
| **Front Door** | Azure | L7 | 全球级；CDN + LB 合一 |

#### 关键功能

- **健康检查 (Health Check)**：间隔（Interval）、超时（Timeout）、健康阈值（Healthy Threshold）、不健康阈值（Unhealthy Threshold）
- **粘性会话 (Sticky Sessions)**：通过 Cookie 将同一用户绑定到同一后端实例（有状态应用）
- **SSL 终止 (SSL Termination)**：LB 处理 HTTPS，后端用 HTTP，减轻后端加密开销
- **连接耗尽 (Connection Draining)**：实例下线前等待现有连接完成，默认 **300 秒**，可配置 0-3600s
- **跨区负载均衡 (Cross-Zone LB)**：流量均匀分配到所有 AZ 的实例（否则按 AZ 均分）

#### 弹性伸缩组件

| 组件 | AWS | GCP | Azure |
|------|-----|-----|-------|
| 实例模板 | Launch Template | Instance Template | VM Image |
| 伸缩组 | **ASG** (Auto Scaling Group) | **MIG** (Managed Instance Group) | **VMSS** (VM Scale Set) |

#### 伸缩策略类型

| 策略 | 描述 |
|------|------|
| **Target Tracking（目标跟踪）** | 维持指标在目标值（如 CPU 50%），最常用 |
| **Step Scaling（阶梯伸缩）** | 按告警幅度分阶段伸缩 |
| **Scheduled Scaling（定时伸缩）** | 按时间表预先伸缩（Black Friday 前扩容） |
| **Predictive Scaling（预测伸缩）** | 基于历史数据 ML 预测，提前扩容 |

**关键时间参数**：
- **冷却时间 (Cooldown)**：伸缩操作后的等待期，默认 **300 秒**（防止频繁伸缩）
- **预热时间 (Warm-up Time)**：新实例就绪前的缓冲时间（不计入指标）

## 对比表格

### SPF vs DKIM vs DMARC

| 维度 | SPF | DKIM | DMARC |
|------|-----|------|-------|
| 机制 | IP 白名单 | 数字签名 | 策略对齐 |
| 防转发失效 | ❌ 失效 | ✅ 有效 | 依赖底层 |
| DNS 存储 | TXT 记录 | TXT 记录（公钥） | TXT 记录 |
| 单独有效 | 部分 | 部分 | 需要 SPF 或 DKIM |
| 执行动作 | 无（建议拒绝） | 无 | none/quarantine/reject |

### L4 vs L7 负载均衡

| 维度 | L4 (NLB) | L7 (ALB) |
|------|----------|----------|
| 协议感知 | TCP/UDP | HTTP/HTTPS |

| 性能 | 极高（百万 RPS） | 高（需解析 HTTP） |

| WebSocket | ✅ | ✅ |
| 内容路由 | ❌ | ✅ |

### ASG vs MIG vs VMSS

| 维度 | AWS ASG | GCP MIG | Azure VMSS |
|------|---------|---------|-----------|
| 配置来源 | Launch Template | Instance Template | VM Image + Config |
| 健康检查 | ELB + EC2 Health | LB + VM Health | LB + VM Health |
| 自动修复 | ✅ | ✅ | ✅ |
| 预测伸缩 | ✅ (Predictive) | 部分支持 | ✅ |

## ⭐ 易考点

1. **CNAME 不能用于 Zone Apex**（如裸域 `example.com`），必须用 A 记录或 AWS Alias 记录

2. **SPF 在邮件转发时失效**，因为转发改变了 Envelope From 的 IP；DKIM 签名在转发后依然有效

3. **Error Budget = 1 - SLO**；SLO 99.9% → 每月约 43.8 分钟停机预算；预算耗尽则冻结新功能

4. **S3 所有存储层级共享 11个9 耐久性**；存储类影响的是可用性、检索延迟和成本，不影响耐久性

5. **连接耗尽 (Connection Draining) 默认 300 秒**；ALB 叫 Deregistration Delay，NLB 叫 Connection Draining；设为 0 表示立即断开

6. **日志禁止记录 PII**（密码、邮箱、信用卡号）；时间戳必须用 UTC；生产环境不应开启 DEBUG 级别

7. **GCP HTTP(S) LB 是全球 Anycast**（单个全球 VIP），而 AWS ALB 是区域级（每个区域独立）；GCP Cloud CDN 必须绑定 HTTP(S) LB

## 自测题

### 多选题 (Multiple Select)

**Q1. 关于 DNS 记录类型，以下哪些说法正确？**

A. A 记录将域名映射到 IPv4 地址  
B. CNAME 记录可以用于裸域（Zone Apex，如 `example.com`）  
C. MX 记录包含邮件服务器的优先级字段  
D. TXT 记录常用于存储 SPF 和 DKIM 公钥  
E. NS 记录指定了该区域的授权名称服务器  

<details>
<summary>答案与解析</summary>

**正确答案：A、C、D、E**

- **A ✅**：A 记录是最基础的域名→IPv4 映射记录
- **B ❌**：CNAME 不能用于 Zone Apex，因为 Zone Apex 必须有 SOA 和 NS 记录，CNAME 不能与其他记录共存。AWS Route 53 的 Alias 记录可以解决这个问题
- **C ✅**：MX 记录格式为 `优先级 邮件服务器域名`，数字越小优先级越高
- **D ✅**：SPF 存储在 TXT 记录中；DKIM 的**公钥**也发布在 TXT 记录中（私钥保留在发件方服务器）
- **E ✅**：NS 记录定义了谁是该 Zone 的权威答复者

</details>

**Q2. 以下哪些是 AWS 弹性伸缩组（ASG）支持的伸缩策略类型？**

A. Target Tracking Scaling（目标跟踪）  
B. Step Scaling（阶梯伸缩）  
C. Scheduled Scaling（定时伸缩）  
D. Predictive Scaling（预测伸缩）  
E. Reactive Scaling（响应伸缩）  

<details>
<summary>答案与解析</summary>

**正确答案：A、B、C、D**

- **A ✅**：目标跟踪最常用，例如维持平均 CPU 使用率在 50%
- **B ✅**：阶梯伸缩根据告警触发幅度执行不同规模的伸缩操作
- **C ✅**：定时伸缩按 cron 表达式在特定时间扩缩容（如 Black Friday 活动前预扩容）
- **D ✅**：预测伸缩使用机器学习分析历史流量模式，提前扩容
- **E ❌**：AWS ASG 中没有名为 "Reactive Scaling" 的策略类型，这是个干扰项

</details>

**Q3. 关于可观测性的日志最佳实践，以下哪些是正确的？**

A. 生产环境应使用 DEBUG 级别以捕获最多信息  
B. 日志不应包含用户密码、信用卡号等 PII 信息  
C. 日志时间戳应使用 UTC 时间，避免时区问题  
D. 结构化 JSON 格式的日志比纯文本更便于机器解析  
E. WARN 级别表示不可恢复的严重错误，应立即告警  

<details>
<summary>答案与解析</summary>

**正确答案：B、C、D**

- **A ❌**：DEBUG 级别产生海量输出，会影响性能并增加存储成本，生产环境通常使用 INFO 或 WARN 级别
- **B ✅**：记录 PII 违反隐私法规（GDPR、CCPA），也是安全漏洞风险
- **C ✅**：分布式系统跨时区部署时，UTC 是唯一无歧义的时间基准
- **D ✅**：JSON 格式可被 Elasticsearch、Datadog 等工具直接解析，支持结构化查询
- **E ❌**：WARN 表示**可恢复**的异常（如重试成功）；不可恢复的严重错误是 ERROR 或 FATAL

</details>

### 判断题 (True/False)

**Q4. DKIM 签名在邮件被转发后会失效，因为转发改变了发件人 IP 地址。**

<details>
<summary>答案与解析</summary>

**判断：False（错误）**

这个描述是 **SPF** 的特性，不是 DKIM 的特性。

- **SPF** 基于 IP 地址验证，当邮件转发时，新的转发服务器 IP 不在原域的 SPF 白名单中，导致 SPF 验证**失败**
- **DKIM** 基于**数字签名**（非对称加密），签名写在邮件头部（`DKIM-Signature`），转发过程通常不修改邮件头，因此 DKIM 签名在转发后**依然有效**

这也是为什么 DMARC 在转发场景下更依赖 DKIM 对齐而非 SPF 对齐。

</details>

**Q5. AWS S3 的 Glacier Deep Archive 存储类与 S3 Standard 的数据耐久性（Durability）相同，都是 99.999999999%（11个9）。**

<details>
<summary>答案与解析</summary>

**判断：True（正确）**

S3 所有存储类别（Standard、Standard-IA、One Zone-IA、Glacier Instant Retrieval、Glacier Flexible Retrieval、Glacier Deep Archive、Intelligent-Tiering）都提供相同的 **11个9（99.999999999%）**耐久性。

不同存储类别的区别在于：
- **可用性 (Availability)**：Standard 是 99.99%，Standard-IA 是 99.9%
- **数据取回延迟**：Standard 毫秒级，Glacier Deep Archive 最长 12 小时
- **存储成本**：Glacier Deep Archive 是最便宜的层级
- **最低存储时长**：Glacier Deep Archive 要求最少 180 天

耐久性保证由 AWS 通过跨多个 AZ 的冗余存储实现，与存储类无关。

</details>

**Q6. GCP HTTP(S) 负载均衡器（HTTP(S) Load Balancer）是全球级的，可以通过单个全球 Anycast IP 地址接受来自世界各地的流量。**

<details>
<summary>答案与解析</summary>

**判断：True（正确）**

GCP HTTP(S) Load Balancer 是**全球级（Global）**负载均衡器，具有以下特点：
- 使用 **Anycast** 路由：全球用一个 VIP（虚拟 IP），BGP 协议自动将请求引导到最近的 Google PoP
- 流量在 Google 全球骨干网内传输，降低跨区延迟
- 可配合 **Cloud CDN** 实现内容缓存（必须绑定到 HTTP(S) LB）

对比：
- **AWS ALB** 是**区域级**的，每个区域部署独立的 ALB，跨区域需要 Route 53 路由策略
- **Azure Front Door** 类似 GCP 的全球 LB 方案（全球 L7 Anycast）

</details>

### 场景题 (Scenario)

**Q7. TechStartup 公司的工程师 Alex 正在设计邮件通知系统，使用 AWS SES 发送用户注册验证邮件和营销邮件。上线一个月后，运营团队发现大量用户反映验证邮件进入垃圾箱，同时 SES 控制台显示投诉率为 0.15%，退信率为 7%。请分析问题原因并给出具体修复方案。**

<details>
<summary>答案与解析</summary>

**问题分析**

当前状态警报：
- **投诉率 0.15%**：超过 0.1% 警告线（接近 0.5% 暂停线）
- **退信率 7%**：超过 5% 警告线（接近 10% 暂停线）

这两个指标都已触发 AWS SES 警告，如不处理，SES 账号将被暂停发送权限。

**根本原因**

1. **营销邮件与事务邮件混发**：营销邮件（用户可能不想收）和关键事务邮件（验证码）用同一发件人/IP 发送，营销邮件产生的投诉污染了整体声誉

2. **邮件列表质量差**：退信率 7% 说明列表中存在大量无效邮箱（硬退信），可能包含垃圾邮件陷阱（Spam Traps）

3. **邮件认证未完整配置**：可能缺少 DMARC 策略，导致收件方无法验证合法性

**修复方案**

**立即行动（降低指标）**：
1. **清洗邮件列表**：删除所有硬退信地址，使用 double opt-in（二次确认）验证新订阅者
2. **停止向未验证地址发营销邮件**：暂停营销邮件，先恢复声誉

**短期（技术修复）**：
3. **分离发送流**：为营销邮件和事务邮件使用**不同的 SES 发送身份**（不同子域名，如 `notify@mail.example.com` vs `promo@promo.example.com`），隔离声誉影响
4. **完整部署邮件认证**：
   - SPF：在 DNS 添加 TXT 记录授权 SES IP
   - DKIM：在 SES 控制台生成密钥对，将公钥发布到 DNS
   - DMARC：先设置 `p=none` 监控模式收集报告，再逐步升级到 `p=quarantine`/`p=reject`

**长期（流程改进）**：
5. **设置 SNS 通知**：订阅 SES Bounce/Complaint 通知，自动将退信/投诉地址加入黑名单
6. **监控健康指标**：在 CloudWatch 配置告警，退信率 >3% 或投诉率 >0.08% 时提前预警

</details>

**Q8. 电商公司的架构师 Maria 负责优化公司全球网站的访问速度。网站静态资源（图片、CSS、JS）托管在 S3，动态 API 运行在 EC2 上，目标是让全球用户都能快速访问，同时确保在促销活动时能自动扩容。请设计完整的架构方案并解释关键技术选型。**

<details>
<summary>答案与解析</summary>

**架构设计方案**

```
全球用户
    │
    ▼
Route 53（DNS + 健康检查 + 地理路由）
    │
    ├──► CloudFront（CDN，全球 400+ PoPs）
    │         │
    │    ┌────┴────────────────────┐
    │    │                         │
    │    ▼                         ▼
    │  S3（静态资源）           ALB（动态 API）
    │  Cache-Control 长 TTL        │
    │                         ASG（弹性伸缩 EC2）
    │                         Launch Template + 伸缩策略
    │
    └──► RDS Multi-AZ（数据库高可用）
```

**关键技术选型解释**

**1. Route 53 + CloudFront（解决全球访问速度）**
- 使用 **Route 53 Latency-Based Routing** 将用户引导到最近区域
- **CloudFront** 在全球 400+ PoPs 缓存静态资源，用户就近获取，大幅降低延迟
- 静态资源设置长 TTL（如 `Cache-Control: max-age=86400`），减少回源
- 版本化 URL（Cache Busting）：部署新版本时文件名加哈希（如 `app.abc123.js`），避免用户获取旧缓存

**2. CloudFront + S3（静态资源分发）**
- S3 存储静态文件，CloudFront 作为 CDN 层
- 禁止用户直接访问 S3（使用 Origin Access Control），强制走 CloudFront

**3. ALB + ASG（动态 API 弹性扩容）**
- **ALB（L7）** 处理 HTTP 请求，支持基于 URL 路径路由（`/api/*` → API 服务器）
- **ASG 伸缩策略**：
  - 平时：Target Tracking（目标 CPU 60%）
  - 促销活动前：Scheduled Scaling 提前扩容（避免冷启动延迟）
  - 流量突增时：Predictive Scaling 基于历史模式预扩容
- **预热时间（Warm-up）**：设置新实例预热时间，避免未就绪实例接收流量
- **连接耗尽（Connection Draining）**：确保缩容时不中断现有用户请求

**4. 缓存失效策略（促销活动时更新价格）**
- 促销开始前，调用 **CloudFront Invalidation API** 主动清除商品页面缓存
- 对频繁变化的动态内容设置短 TTL 或 `Cache-Control: no-cache`

**DNS 配置**：
- CloudFront 域名通过 Route 53 **Alias 记录**绑定（可用于裸域，且免额外 DNS 查询费）

</details>
