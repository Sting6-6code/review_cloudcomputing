## 2. 核心概念

### 2.1 云计算基础（L01）

**On-Premise（本地部署）**：服务器、操作系统、硬件全部在自己的机房内管理，一切自己负责。

**Cloud Computing（云计算）**：NIST 定义——通过网络按需访问共享的、可配置的计算资源池，可快速交付，管理成本极低。包含 **5 大特征、3 种服务模型、4 种部署模型**。

**5 大云特征（NIST）**：
1. **On-demand Self-service（按需自助服务）**：用户无需人工介入即可自行开通资源。
2. **Broad Network Access（广泛网络访问）**：通过标准网络机制从任何设备访问。
3. **Resource Pooling（资源池化 / Multitenancy 多租户）**：提供商将资源池化供多用户共享，通过虚拟化隔离租户。
4. **Elasticity（弹性）**：自动按需扩缩资源，是云的核心价值之一。
5. **Measured Service（计量服务）**：按实际使用量计费（pay-as-you-go）。

**CapEx vs OpEx**：传统 IT 需要前期资本支出（Capital Expenditure），云计算转换为按用量的运营支出（Operating Expenditure）。

**Capacity Planning（容量规划）**：三种策略——
- **Lead Strategy**：提前扩容（避免不足）
- **Lag Strategy**：容量耗尽后再扩（避免浪费）
- **Match Strategy**：随需求小幅增量扩容

> 关键数据：传统服务器平均利用率常低于 20%，大量资源在非峰值时间闲置。

**Scaling（扩展性）**：
- **Horizontal Scaling（水平扩展）**：Scale Out（增加实例）/ Scale In（减少实例）
- **Vertical Scaling（垂直扩展）**：Scale Up（增大单机配置）/ Scale Down（降低配置）

### 2.2 三种服务模型（Service Models）

| 模型 | 你管理的层 | 供应商管理的层 | 典型例子 |
|------|-----------|--------------|---------|
| **IaaS** | App、Data、Runtime、Middleware、OS | 虚拟化、服务器、存储、网络 | AWS EC2、GCP、Azure |
| **PaaS** | App、Data | OS 及以下全部 | Heroku、Google App Engine |
| **SaaS** | 仅用户配置 | 一切 | Gmail、GitHub、Google Drive |

> 记忆口诀（Pizza 类比）：IaaS = 买食材自己烤（Take & Bake）；PaaS = 外卖到家（Pizza Delivered）；SaaS = 去餐厅吃（Dined Out）。

### 2.3 四种部署模型（Deployment Models）

| 模型 | 说明 | 典型场景 |
|------|------|---------|
| **Public Cloud** | 对公众开放，供应商负责维护 | AWS、Azure、GCP |
| **Community Cloud** | 限定特定社区使用，如政府机构 | AWS GovCloud（符合 ITAR/FedRAMP） |
| **Private Cloud** | 专属某一组织，注意：**不等于 On-Prem** | 企业内部私有云 |
| **Hybrid Cloud** | 两种或以上部署模型组合 | 敏感数据私有云 + 普通业务公有云 |

### 2.4 SDLC — 软件开发生命周期（L02_01）

**SDLC（Software Development Lifecycle）**：将软件从概念到上线维护的结构化流程，确保质量和用户需求的系统化方法。

**6 个阶段**（云原生视角）：

| 阶段 | 职责 | 云原生关注点 |
|------|------|------------|
| 1. Planning | 需求、可行性、风险 | 提前规划基础设施成本和服务边界 |
| 2. Design | 架构、数据库、API、安全 | 遵循 12-Factor，特别是无状态和配置外部化 |
| 3. Implementation | 编码、版本控制、单元测试 | 代码需容器就绪（container-ready） |
| 4. Testing | 单元、集成、系统、验收测试 | 包含容器编排和混沌工程（Chaos Engineering） |
| 5. Deployment | 发布到生产环境 | CI/CD pipeline 和 IaC 是必需的 |
| 6. Maintenance | 监控、事故响应、性能调优 | 可观测性（Observability）：日志、指标、追踪 |

**三种 SDLC 模型对比**：

| 维度 | Waterfall（瀑布） | Agile（敏捷） | DevOps |
|------|----------------|-------------|--------|
| 流程 | 线性顺序，不可回退 | 迭代式冲刺（Sprint） | 持续流，无离散边界 |
| 变更响应 | 极难响应变更 | 拥抱变更 | 自动化变更流水线 |
| 测试时机 | 开发完成后 | 每个迭代中 | 全程自动化（Shift-Left） |
| 适用场景 | 需求稳定、合规驱动 | 需求不确定、快速迭代 | 高频发布、微服务 |

**云原生 SDLC 原则**：
- **Shift-Left**：将测试和安全扫描提前到开发阶段，而非等到上线前。
- **GitOps**：Git 作为代码和基础设施的唯一真相来源，变更通过 PR + 自动化验证流入。
- **Independent Service Lifecycles**：每个微服务有独立的 SDLC 节奏，团队端到端负责。

### 2.5 12-Factor App（L02_02）

**起源**：Heroku 工程师于 2011 年总结的云原生应用设计方法论，解决可移植性、可扩展性和持续交付问题。

| # | Factor | 核心要点 |
|---|--------|---------|
| 1 | **Codebase** | 一个仓库，多个部署环境（dev/staging/prod） |
| 2 | **Dependencies** | 显式声明依赖（requirements.txt、package.json），不依赖系统全局包 |
| 3 | **Config** | 配置存环境变量，**绝不**将凭证或环境特定值写入代码 |
| 4 | **Backing Services** | 数据库、队列、SMTP 等均视为"附加资源"，可通过配置替换 |
| 5 | **Build, Release, Run** | 严格分离：Build（编译）→ Release（与配置合并）→ Run（执行） |
| 6 | **Processes** | 进程无状态（Stateless）且 Share-Nothing，Session 存外部数据库 |
| 7 | **Port Binding** | 应用自身内嵌 Web Server，通过端口暴露服务 |
| 8 | **Concurrency** | 通过进程模型水平扩展（Web 进程、Worker 进程分类型） |
| 9 | **Disposability** | 快速启动 + 优雅关闭（SIGTERM），增强弹性和可扩展性 |
| 10 | **Dev/Prod Parity** | 开发、测试、生产环境尽量一致，避免"在我这里没问题" |
| 11 | **Logs** | 日志视为事件流，写 stdout，由平台收集，应用不管理日志文件 |
| 12 | **Admin Processes** | 数据库迁移等管理任务作为一次性进程运行，使用相同代码和配置 |

### 2.6 DevOps（L02_03）

**DevOps**：打破开发（Dev）和运维（Ops）之间的壁垒，通过文化、实践和工具提升高频交付能力。

**历史背景**：传统模式下，开发写完代码"扔过墙"给运维，导致发布慢、责任推诿、生产与开发脱节。DevOps 兴起于 2000 年代末。

**DevOps 三要道（The Three Ways）**：
1. **Flow（流动）**：优化整体系统流动，减少批次大小，消除瓶颈，让工作从开发顺畅流向客户。
2. **Feedback（反馈）**：在所有阶段建立快速反馈机制——自动化测试、监控告警、无责事后分析（Blameless Post-mortem）。
3. **Continuous Learning（持续学习）**：营造试验文化，注入故障以建立韧性（混沌工程），将改进制度化。

**DevOps 核心实践**：

| 实践 | 说明 |
|------|------|
| Infrastructure as Code (IaC) | 基础设施用代码管理，存 Git，与应用代码同等对待 |
| Continuous Integration (CI) | 开发者频繁合并代码，自动触发构建和测试 |
| Continuous Delivery | 代码自动构建测试到可部署状态，需手动批准才发布生产 |
| Continuous Deployment | 全自动，代码通过测试即自动发布生产 |
| Monitoring & Observability | 日志、指标、告警、追踪，为反馈闭环提供数据 |
| Security (DevSecOps) | 安全贯穿整个生命周期，不是最后才加 |

**DORA 指标（衡量 DevOps 成熟度）**：
- **Deployment Frequency**：多久部署一次到生产？
- **Lead Time for Changes**：从代码提交到运行生产需多久？
- **Change Failure Rate**：多少比例的部署导致需要修复的故障？
- **Time to Restore Service**：发生故障后多久恢复？

## 4. 对比表格

### Waterfall vs Agile vs DevOps

| | Waterfall | Agile | DevOps |
|--|-----------|-------|--------|
| 发布周期 | 数月~数年 | 数周（Sprint） | 数小时~数天 |
| 需求变更 | 极难 | 拥抱 | 持续响应 |
| 测试 | 阶段末 | 每个Sprint | 全程自动化 |
| 团队结构 | 职能竖井 | 跨职能小团队 | Dev+Ops融合 |
| 适用场景 | 需求固定、强合规 | 需求模糊、快速验证 | 高频部署、微服务 |

### IaaS vs PaaS vs SaaS

| | IaaS | PaaS | SaaS |
|--|------|------|------|
| 控制粒度 | 最高（OS以上全自管） | 中（只管App和Data） | 最低 |
| 运维负担 | 最重 | 中等 | 最轻 |
| 灵活性 | 最高 | 中等 | 最低 |
| 典型用途 | 自定义基础设施 | 快速部署应用 | 直接使用软件产品 |

### Horizontal vs Vertical Scaling

| | Horizontal（水平）| Vertical（垂直）|
|--|-----------------|----------------|
| 方向 | Scale Out / In（增减实例数） | Scale Up / Down（增减单机配置） |
| 上限 | 理论无上限 | 单机硬件上限 |
| 停机 | 不需要 | 通常需要 |
| 12-Factor | Factor 8（Concurrency）支持 | 不符合云原生理念 |
| 适合 | 无状态应用、高并发 | 数据库等有状态服务 |

### Continuous Integration vs Delivery vs Deployment

| | CI | Continuous Delivery | Continuous Deployment |
|--|----|--------------------|----------------------|
| 自动化止于 | 测试完成 | 准备好部署（需人工批准） | 直接部署到生产 |
| 人工介入 | 代码合并触发 | 最后批准需人工 | 无 |
| 风险程度 | 低 | 中 | 高（需充分测试覆盖） |

## 5. 关键配置/示例

```bash
# 12-Factor Factor 3：配置存环境变量示例
export DATABASE_URL="postgres://user:pass@host:5432/mydb"
export API_KEY="my-secret-key"
export LOG_LEVEL="info"

# 应用内读取（Python示例）
import os
db_url = os.environ['DATABASE_URL']
```

```bash
# 12-Factor Factor 11：日志输出到 stdout（不写文件）
# 正确做法：
print("user logged in", file=sys.stdout)

# 错误做法：
with open('/var/log/app.log', 'a') as f:
    f.write("user logged in")
```

```bash
# 12-Factor Factor 5：Build/Release/Run 分离
# Build阶段
docker build -t myapp:v1.2.3 .

# Release阶段（注入配置）
docker run -e DATABASE_URL=$DB_URL myapp:v1.2.3

# Run阶段由平台管理，Release不可变
```

## 6. ⭐ 易考点

### ⭐ 1. NIST 5大云特征（必背）
考试最爱考"以下哪些是云的基本特征"多选题：**On-demand Self-service / Broad Network Access / Resource Pooling / Elasticity / Measured Service**。

### ⭐ 2. IaaS / PaaS / SaaS 的责任边界
谁管什么层？IaaS 你管 App+Data+Runtime+OS；PaaS 你只管 App+Data；SaaS 你什么都不管。这是场景题高频考点。

### ⭐ 3. Continuous Delivery vs Continuous Deployment 的区别
前者需要人工批准才上线，后者全自动。两词很像，极易混淆，判断题必考。

### ⭐ 4. 12-Factor Factor 3（Config）和 Factor 6（Processes）
配置不能写死在代码里（要用环境变量）；进程必须无状态（Session 不能存进程内存）。这两条是场景题最常考的设计决策。

### ⭐ 5. Private Cloud ≠ On-Premise
私有云可以由组织自己管理，但它仍然是"云"，具备云的弹性和自动化特征。On-Prem 是纯自管机房，两者不同。判断题陷阱！

### 判断题 2

**陈述：** Continuous Delivery（持续交付）和 Continuous Deployment（持续部署）是同一回事，代码通过测试后都会自动部署到生产环境。

**True / False？请解释原因。**

<details>
<summary>查看答案</summary>

**False（错误）**

**解析：**

| | Continuous Delivery | Continuous Deployment |
|--|--------------------|-----------------------|
| 共同点 | 代码自动构建 + 自动测试 | 代码自动构建 + 自动测试 |
| 区别 | 准备好发布状态，但**需要人工批准**才能部署生产 | 通过测试后**全自动**部署到生产，无需人工介入 |

Continuous Delivery 给了团队"可以随时发布"的能力，但保留了最后一道人工门控（通常出于合规、业务节奏等原因）。Continuous Deployment 是更激进的形式，要求极高的测试覆盖率和自动化成熟度。

**记忆技巧**：Delivery（交付）= 送到门口，你还要决定要不要进来；Deployment（部署）= 直接送进来放好。
</details>

### 场景分析题

**场景：**

Marcus 是一家在线教育平台的首席架构师，他们的应用目前是传统单体架构，部署在自己的机房里。平台在每年的大学开学季（9月）会遭遇 10 倍流量峰值，平时服务器利用率只有 15%。最近公司决定迁移到云端，并开始重构为云原生架构。

技术团队正在争论两个问题：

1. **扩展策略**：是应该在开学前临时购置更多服务器（Lead Strategy），还是应该迁移到云后利用弹性自动扩缩？
2. **应用设计**：开发团队习惯在代码中硬编码数据库连接字符串（如 `db_host = "10.0.0.5"`），并将用户 Session 存储在应用服务器的内存中。

请分析：
- (a) 为什么传统方案（购置服务器）在成本上效率低下？
- (b) 迁移到云后，哪种扩展类型更适合应对流量峰值？
- (c) 硬编码配置和内存存 Session 各违反了 12-Factor 的哪条原则？会导致什么问题？

<details>

**(a) 传统方案的成本问题：**

根据课件，传统容量规划的核心矛盾是：**为峰值配置的资源，在非峰值时大量闲置**。Marcus 的平台平均利用率只有 15%，意味着 85% 的服务器资源在非开学季是浪费的（CapEx 已支出但价值未体现）。

选择 Lead Strategy（提前扩容）意味着全年为 9 月峰值买单，而 10 个月里这些服务器几乎空转。这就是云计算要解决的核心问题：**将 CapEx 转换为 OpEx，按需付费**。

**(b) 扩展策略选择：**

迁移到云后，应使用**水平扩展（Horizontal Scaling）+ 弹性（Elasticity）** 组合：

- 配置 **Auto Scaling**（自动扩缩），根据 CPU、请求数等指标在开学季自动增加实例（Scale Out），平时自动缩减（Scale In）
- 这是云 Elasticity 特征的核心价值：按需扩缩，不再为峰值预购资源
- 比垂直扩展（买更大的机器）更灵活，无停机，且符合 12-Factor Factor 8（Concurrency）

**(c) 代码问题与 12-Factor 违反：**

**硬编码数据库连接字符串** → 违反 **Factor 3: Config**
- 问题：同一份代码无法在 dev/staging/prod 间切换，改环境必须改代码后重新构建
- 正确做法：`db_host = os.environ['DB_HOST']`，通过环境变量注入

**Session 存应用服务器内存** → 违反 **Factor 6: Processes**（Stateless Processes）
- 问题：水平扩展时，用户 A 的请求可能到达实例 1（有 Session），下一个请求路由到实例 2（没有 Session），导致用户频繁被踢出登录
- 正确做法：Session 存 Redis 或数据库等外部 Backing Service，进程本身保持无状态，任何实例都能处理任何请求

**总结**：这两个设计缺陷会直接阻碍水平扩展，即使迁移到云上也无法发挥弹性优势。**云原生不是迁移，是重构设计理念。**
