# L06 — IAM、虚拟化、机器镜像与 Cloud-Init

## 模块总览

本模块构建从**身份控制**到**计算实例启动**的完整链路：IAM 决定"谁能访问什么"，虚拟化技术提供弹性计算能力，AMI 作为可复用的镜像模板，Cloud-Init 在首次启动时完成实例个性化配置。

---

## 核心概念

### 一、身份与访问管理（Identity and Access Management, IAM）

#### 1. 核心问题：认证 vs 授权

| 问题 | 术语 | 含义 |
|------|------|------|
| "你是谁？" | **认证（Authentication）** | 证明身份（密码、MFA、密钥） |
| "你能做什么？" | **授权（Authorization）** | 基于身份授予权限 |

**默认拒绝（Secure by Default / Deny by Default）**：云 IAM 中，用户默认对任何资源都没有访问权限，所有权限必须**显式授予**。

**最小权限原则（Principle of Least Privilege）**：只授予完成任务所需的最小权限，不多一分。这是云安全最重要的设计原则之一——IAM 配置错误是云安全泄露最常见的原因。

#### 2. AWS IAM 四大核心组件

| 组件 | 定义 | 凭证类型 | 使用场景 |
|------|------|----------|----------|
| **用户（IAM User）** | 具有长期凭证的个人身份 | 用户名/密码、Access Key | 人员日常操作 |
| **用户组（IAM Group）** | 共享相同权限的用户集合 | — | 批量权限管理（如"开发者组"、"管理员组"）|
| **角色（IAM Role）** | 可被临时"扮演"的权限集合，凭证**有时间限制** | 临时凭证（STS） | 应用程序、EC2 实例、跨账户访问 |
| **策略（IAM Policy）** | JSON 文档，定义允许/拒绝的操作和资源 | — | 权限规则的具体描述 |

> ⭐ **关键区分：Role vs User**
> - User：长期凭证（Access Key 可永久存在），适合人员
> - Role：**临时凭证**，EC2 实例通过 Instance Profile 扮演角色访问其他 AWS 服务，**永远不要把 Access Key 放在 EC2 实例上**

**IAM Policy 结构**（JSON 示例）：
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",           // Allow 或 Deny
      "Action": "s3:GetObject",   // 允许的操作
      "Resource": "arn:aws:s3:::my-bucket/*"  // 作用的资源（ARN）
    }
  ]
}
```

#### 3. AWS IAM 特点
- **全局服务（Global Service）**：不区分 Region，IAM 配置全账户生效
- **身份联合（Identity Federation）**：支持企业 Active Directory（SAML）、Google/Facebook（OIDC）登录，用户无需创建单独的 IAM 账户

#### 4. AWS IAM 最佳实践
- 锁定并极少使用 Root 账户（根账号）
- 为每个人创建独立 IAM 用户，**不共享凭证**
- 使用**用户组（Group）**分配权限，而非直接给用户
- EC2 应用程序使用**角色（Role）**，而非 Access Key
- 为特权用户启用 **MFA（多因素认证）**
- 定期轮换（Rotate）凭证，删除不需要的凭证

#### 5. 三大云 IAM 跨平台对比

| 概念 | AWS | GCP | Azure |
|------|-----|-----|-------|
| 身份提供商 | AWS IAM | Google Cloud Identity | Microsoft Entra ID |
| 人员身份 | IAM Users | Google Accounts | Entra ID Users |
| **机器身份** | IAM Roles / Instance Profiles | **Service Accounts** | **Managed Identities** |
| 权限模型 | JSON Policy（直接绑定）| Role Bindings（资源层级继承）| RBAC Role Assignments（层级继承）|
| 资源层级 | 账户 → Organizations | Org → Folders → Projects | 管理组 → 订阅 → 资源组 |
| 权限继承 | 策略直接附加 | **向下继承**（Project 权限覆盖所有资源）| **向下继承** |
| 默认访问 | **全部拒绝** | **全部拒绝** | **全部拒绝** |

### 二、虚拟化（Virtualization）

#### 1. 什么是虚拟化？

用软件模拟硬件的存在，在单台物理服务器上运行多个虚拟机（Virtual Machine, VM），实现**服务器整合（Server Consolidation）**。

传统物理服务器利用率仅 **10–15%**，虚拟化大幅提升资源利用率。

> **关键区分：虚拟化 ≠ 云计算**
> - 虚拟化是**底层支撑技术**
> - 云计算是**建立在虚拟化之上的交付模型**

#### 2. Hypervisor（虚拟机监控器）

| 类型 | 特征 | 代表产品 | 用途 |
|------|------|----------|------|
| **Type 1（裸金属）** | 直接运行在物理硬件上 | KVM, Xen, Hyper-V, VMware ESXi | **所有主流云提供商使用** |
| **Type 2（托管型）** | 运行在宿主操作系统之上 | VirtualBox, VMware Workstation, Parallels | 本地开发/测试 |

#### 3. 三种虚拟化技术

| 技术 | 客户 OS 是否知道虚拟化？ | 性能 | 现代用法 |
|------|------------------------|------|----------|
| 全虚拟化（Full Virtualization） | ❌ 不知道 | 较低（需翻译所有硬件调用）| 旧 |
| 半虚拟化（Paravirtualization, PV） | ✅ 知道，使用"超级调用（Hypercalls）"| 较好（需修改客户 OS 内核）| 旧（Xen PV）|
| **硬件辅助虚拟化（Hardware-Assisted）** | ❌ 不知道（CPU 原生支持） | **接近原生性能** | **现代云主流（Intel VT-x / AMD-V）** |

#### 4. VM vs 容器（Containers）

| 维度 | 虚拟机（VM） | 容器（Container）|
|------|------------|-----------------|
| 虚拟化层次 | 虚拟化**硬件** | 虚拟化**操作系统** |
| 隔离级别 | 强（Hypervisor 隔离） | 弱（共享宿主内核） |
| 启动时间 | 分钟级 | 秒级（毫秒级）|
| 资源开销 | 大（每个 VM 含完整 OS）| 小 |
| 典型工具 | EC2, GCE | Docker, Kubernetes |

#### 5. EC2 实例类型（Instance Types）

| 类型 | 特点 | 典型实例族 | 适用场景 |
|------|------|-----------|----------|
| 通用型（General Purpose） | CPU/内存/网络均衡 | t3, m6i | Web 服务器、开发环境 |
| 计算优化型（Compute Optimized） | 高性能 CPU | c6i | 批处理、游戏服务器 |
| 内存优化型（Memory Optimized） | 大内存 | r6i | 数据库、内存缓存 |
| 加速计算型（Accelerated/GPU） | GPU/FPGA | p4d, g5 | 机器学习、HPC |
| 存储优化型（Storage Optimized） | 高 IOPS 本地存储 | i3, d3 | 数据仓库、Hadoop |

#### 6. EC2 定价模型

| 定价模式 | 折扣 | 承诺 | 适用场景 |
|----------|------|------|----------|
| **按需（On-Demand）** | 无折扣（基准价）| 无 | 短期、不规律、不可中断 |
| **预留实例（Reserved）** | 最高 **75%** | 1年或3年 | 稳定、持续运行的工作负载 |
| **竞价实例（Spot）** | 最高 **90%** | 无（可被中断）| 弹性、可中断、批量任务 |
| **专用主机（Dedicated Host）** | — | 按主机计费 | BYOL（自带许可证）、合规要求 |
| **专用实例（Dedicated Instance）** | — | 按实例 + **$2/区域附加费** | 物理隔离但不需要完整主机 |

GCP 特有折扣：
- **持续使用折扣（Sustained Use Discounts, SUDs）**：实例运行超过月份的 **25%** 自动享受最高 **30%** 折扣，无需承诺
- **承诺使用折扣（Committed Use Discounts, CUDs）**：1年或3年承诺，最高 **70%** 折扣
- **竞价虚拟机（Spot VMs）**：自动折扣 **60–91%**

#### 7. Meltdown & Spectre（安全漏洞）

- **Meltdown（CVE-2017-5754）**：利用 Intel CPU 乱序执行漏洞，允许用户态程序读取内核内存。**仅影响 Intel（AMD 不受影响）**。
- **Spectre（CVE-2017-5753/5715）**：利用推测执行（Speculative Execution）漏洞，影响 Intel、AMD、ARM 几乎所有现代 CPU。
- **云场景风险**：多租户环境中，恶意 VM 可能读取同一物理主机上其他租户的数据。
- **修复代价**：内核页表隔离（KPTI）引入 **5–30%** 性能损耗；Epic Games 修补后 CPU 利用率上升约 **300%**。

### 三、机器镜像（Machine Images）

#### 1. 什么是 AMI（Amazon Machine Image）？

AMI 是启动 EC2 实例所需信息的**模板**，包含三个组件：
1. **根卷模板（Root Volume Template）**：操作系统、应用服务器、应用程序
2. **启动权限（Launch Permissions）**：控制哪些 AWS 账户可以使用该 AMI
3. **块设备映射（Block Device Mapping）**：指定启动时附加哪些 EBS 卷或实例存储卷

#### 2. Instance Store-Backed vs EBS-Backed AMI（⭐高频考点）

| 维度 | Instance Store-Backed | **EBS-Backed**（现代标准）|
|------|----------------------|--------------------------|
| 根卷存储位置 | 物理主机上的临时存储（Ephemeral） | EBS 快照（Snapshot）|
| 实例停止（Stop）| **❌ 不支持 Stop**，只能 Running 或 Terminated | ✅ 支持 Stop/Start |
| 数据持久性 | 实例终止/故障后数据**永久丢失** | 停止后数据**保留**（EBS 独立于实例存在）|
| AMI 存储位置 | Amazon S3 | EBS Snapshot |
| 启动速度 | **慢**（需从 S3 复制完整镜像）| **快**（只需加载启动所需部分）|
| 典型用途 | 临时批处理、无状态计算 | 几乎所有现代 EC2 实例 |

#### 3. AMI 生命周期

```
Create（创建）
    → 从运行或停止的实例创建，EBS-backed 实例会自动创建快照
    ↓
Register（注册）
    → AMI 获得唯一 ID（如 ami-0abcdef1234567890）
    ↓
Copy（复制）
    → AMI 是区域（Regional）资源，跨区域需显式 Copy
    → 复制时不传递标签、权限；可加密
    ↓
Share（共享）
    → 默认私有；可共享给特定账户；可公开
    → 加密卷的 AMI 不能公开，只能共享给特定账户
    ↓
Deprecate（弃用）
    → 标记为过时，仍可通过 AMI ID 使用，但隐藏在列表中
    ↓
Disable（禁用）
    → 临时阻止使用，可重新启用
    ↓
Deregister（注销/删除）
    → 永久删除 AMI，但关联的 EBS 快照需单独删除！
    → 不影响已运行的实例
```

> ⚠️ **易错点**：Deregister AMI 后，底层 EBS 快照**不会**自动删除，需要手动清理，否则持续产生存储费用。

#### 4. HVM vs PV AMI

| 类型 | 特征 | AWS 建议 |
|------|------|----------|
| **HVM（Hardware Virtual Machine）** | 完整虚拟化硬件，可利用 CPU 硬件加速、增强网络、GPU | **所有当前代实例类型必须用 HVM** |
| PV（Paravirtual） | 旧版半虚拟化方式 | 遗留系统，**不推荐新实例** |

#### 5. 黄金镜像（Golden Image）

组织内所有实例的**规范起点**：包含已打补丁的 OS、安全加固、监控代理、日志配置和组织标准。

**分层策略**：
```
Layer 0：云提供商基础镜像（Amazon Linux 2023, Ubuntu 24.04）
    ↓
Layer 1：组织黄金镜像（OS 加固 + 安全代理 + 日志配置）
    ↓
Layer 2：角色专用镜像（运行时：Java/Node.js/Python + Nginx）
    ↓
Layer 3：应用镜像（你的应用代码 + 配置文件）
```

安全补丁只需更新 Layer 1，然后逐层重建，大幅减少维护工作量。

#### 6. Packer（镜像自动化构建工具）

HashiCorp Packer 是自动化机器镜像构建工具，使用 HCL2 模板（`.pkr.hcl`）：

```hcl
packer {
  required_plugins {
    amazon = {
      source  = "github.com/hashicorp/amazon"
      version = "~> 1"
    }
  }
}

source "amazon-ebs" "webapp" {
  ami_name      = "csye6225-webapp-{{timestamp}}"
  instance_type = "t2.micro"
  region        = "us-east-1"
  source_ami_filter {
    filters = {
      name                = "al2023-ami-*-x86_64"
      root-device-type    = "ebs"
      virtualization-type = "hvm"
    }
    most_recent = true
    owners      = ["amazon"]
  }
  ssh_username = "ec2-user"
}

build {
  sources = ["source.amazon-ebs.webapp"]
  provisioner "shell" {
    scripts = [
      "scripts/update-os.sh",
      "scripts/install-app.sh",
      "scripts/cleanup.sh"
    ]
  }
}
```

**Packer 在 CI/CD 中的工作流**：
```
代码提交
    → packer init（下载插件）
    → packer validate（检查语法）
    → packer build（启动临时实例 → 执行脚本 → 创建 AMI → 终止临时实例）
    → 输出 AMI ID → 传入 Terraform → 部署新实例
```

### 四、Cloud-Init

#### 1. 什么是 Cloud-Init？

Cloud-Init 是云虚拟机**首次启动时**自动完成初始化的行业标准工具，由 Canonical（Ubuntu 背后的公司）开发，现支持几乎所有主流 Linux 发行版和云提供商。

**作用**：弥合静态镜像与动态配置之间的差距——镜像提供通用底座，Cloud-Init 注入实例专属信息。

#### 2. 镜像无法提供、需要 Cloud-Init 完成的事项

- 实例专属的 SSH 授权密钥（每次部署可能不同）
- 环境专属配置（数据库 URL、API 端点、Feature Flags，dev/prod 不同）
- 密钥和凭证（不应烘焙进镜像，镜像可能被共享或长期存储）
- 实例自身的身份（主机名、IP、在集群中的角色）

#### 3. Cloud-Init 五个启动阶段

| 阶段 | 名称 | 时机 | 作用 |
|------|------|------|------|
| 1 | Generator | 网络前 | 判断是否需要运行 Cloud-Init |
| 2 | Local（cloud-init-local） | 网络前 | 应用网络配置 |
| 3 | Network（cloud-init） | 网络就绪后 | 从 IMDS 获取元数据和 User Data |
| 4 | Config（cloud-config） | — | 运行配置模块（安装包、创建用户） |
| 5 | Final（cloud-final） | — | 运行用户脚本和最终模块 |

#### 4. 数据来源（Data Sources）

Cloud-Init 从**实例元数据服务（Instance Metadata Service, IMDS）**获取指令：
- **AWS**：`http://169.254.169.254/latest/meta-data/`
- **GCP**：`http://metadata.google.internal/computeMetadata/v1/`

**User Data**（自定义配置）传入方式：
- AWS：EC2 启动时的 "User data" 字段
- GCP：Startup script 或自定义 metadata 字段

#### 5. User Data 格式

```bash
# 格式1：Cloud-config YAML（推荐，声明式）
# 必须以 #cloud-config 开头
#cloud-config
package_update: true
package_upgrade: true
packages:
  - nginx
  - curl
  - git

users:
  - name: deploy
    groups: sudo
    shell: /bin/bash
    ssh_authorized_keys:
      - ssh-rsa AAAA...（公钥内容）

write_files:
  - path: /etc/app/config.env
    content: |
      DATABASE_URL=postgres://prod-db:5432/mydb
      APP_ENV=production
    permissions: '0600'

runcmd:
  - systemctl enable nginx
  - systemctl start nginx
  - /opt/app/start.sh
```

```bash
# 格式2：Shell 脚本（命令式）
#!/bin/bash
apt-get update -y
apt-get install -y nginx
systemctl start nginx
```

#### 6. 调试 Cloud-Init

| 日志文件 | 用途 |
|----------|------|
| `/var/log/cloud-init.log` | Cloud-Init 每个动作的详细日志，**首要调试资源** |
| `/var/log/cloud-init-output.log` | 捕获 runcmd 命令和脚本的 stdout/stderr |
| `/run/cloud-init/` | 运行时状态和状态信息 |

**常见坑**：
- YAML 缩进错误（最常见原因）
- 缺少 `#cloud-config` 头部（会被静默忽略或当 shell 脚本执行）
- 误以为 User Data 在每次重启都执行——**默认只在首次启动执行一次**

#### 7. 烘焙进镜像 vs 用 Cloud-Init 注入

| 放哪里 | 原则 | 例子 |
|--------|------|------|
| **烘焙进镜像（Bake）** | 变化慢、所有实例共享 | OS 包、运行时（Java/Node）、监控代理、系统加固 |
| **Cloud-Init 注入** | 动态、实例专属、敏感 | SSH 密钥、数据库 URL、主机名、从 Secrets Manager 拉取的密钥 |

**经验法则**：安装编译需要超过 2-3 分钟的 → 烘焙进镜像。包含密钥或因环境而异的 → 用 Cloud-Init。

## 概念间关系

```
IAM（身份与权限控制）
    ↓ 决定谁有权启动实例、访问资源
虚拟化（Virtualization）
    ↓ Type 1 Hypervisor 在物理机上运行多个 VM，提供弹性计算
EC2 实例（Virtual Machine）
    ↑ 从 AMI 启动
AMI（机器镜像）
    ← Packer 自动构建（CI/CD 触发）
    ← 黄金镜像分层（OS → 组织 → 角色 → 应用）
    ↓ 实例启动时 Cloud-Init 读取 User Data
Cloud-Init
    ↓ 注入动态配置（SSH 密钥、环境变量、服务启动）
运行中的 EC2 实例（生产就绪）

存储选择：
    EBS（Elastic Block Store）← 持久化，支持 Stop/Start，现代标准
    Instance Store          ← 临时存储，高 IOPS，实例终止数据丢失

完整自动化流水线：
    IAM Role 授权 → Packer 构建 AMI → Terraform 部署 → Cloud-Init 初始化
```

## 对比表格

### 1. Instance Store vs EBS（⭐高频考点）

| 维度 | Instance Store（实例存储）| EBS（弹性块存储）|
|------|--------------------------|-----------------|
| 存储位置 | 物理主机本地磁盘（临时）| 独立网络存储服务 |
| 数据持久性 | **临时（Ephemeral）**：实例停止/终止/故障数据全丢 | **持久**：独立于实例生命周期 |
| 支持 Stop 操作 | **❌ 不支持** | ✅ 支持 |
| 读写性能 | **极高**（本地磁盘，低延迟）| 高（网络存储，略低于本地）|
| 是否额外计费 | 包含在实例价格中 | 按 GB/月单独计费 |
| 数据共享 | ❌ 不能在实例间共享 | ✅ 可 detach 后 attach 到其他实例 |
| 快照（Snapshot）| ❌ 不支持 | ✅ 支持（可制作 AMI）|
| 典型用途 | 临时缓存、高性能本地 I/O、MapReduce 中间结果 | 根卷、数据库、需要持久化的数据 |

### 2. AMI vs EBS Snapshot

| 维度 | AMI | EBS Snapshot |
|------|-----|-------------|
| 是什么 | EC2 实例的启动模板 | 单个 EBS 卷的备份 |
| 包含内容 | 根卷 + 启动权限 + 块设备映射 | 单个卷的数据 |
| 直接用途 | 启动新 EC2 实例 | 恢复数据、创建 AMI |
| 区域范围 | Regional（需 Copy 跨区）| Regional（需 Copy 跨区）|
| 关系 | EBS-backed AMI 包含 EBS Snapshot | AMI 的底层存储 |

### 3. Type 1 vs Type 2 Hypervisor

| 维度 | Type 1（裸金属，Bare-Metal）| Type 2（托管型，Hosted）|
|------|---------------------------|------------------------|
| 运行在 | 直接在物理硬件上 | 运行在宿主 OS 之上 |
| 性能 | **更高**（无宿主 OS 开销）| 较低 |
| 隔离性 | **更强** | 较弱 |
| 代表产品 | KVM, Xen, VMware ESXi, Hyper-V | VirtualBox, Parallels, VMware Workstation |
| 使用场景 | **所有主流云提供商**（AWS/GCP/Azure）| 本地开发、测试 |

### 4. 可变基础设施 vs 不可变基础设施（Mutable vs Immutable Infrastructure）

| 维度 | 可变（Mutable）| 不可变（Immutable）|
|------|--------------|------------------|
| 变更方式 | SSH 登录修改运行中服务器 | 构建新 AMI，替换旧实例 |
| 配置漂移 | ❌ 高风险 | ✅ 消除 |
| 回滚 | 复杂（反向操作）| 简单（启动旧版 AMI）|
| 工具 | Ansible, Chef, Puppet | Packer + AMI + Terraform |
| 云原生选择 | 不推荐 | **推荐** |

### 5. IAM Role vs IAM User

| 维度 | IAM Role | IAM User |
|------|----------|----------|
| 身份类型 | 服务/应用/跨账户 | 人员 |
| 凭证类型 | **临时凭证（STS，有过期时间）** | 长期凭证（Access Key 不过期）|
| 安全性 | **更高**（凭证自动轮换）| 较低（长期 Key 若泄露风险大）|
| 推荐用于 | EC2 实例访问其他 AWS 服务 | 人员操作（+MFA）|
| EC2 使用方式 | 通过 Instance Profile 关联 | ❌ 不应将 Access Key 放在实例上 |

## ⭐ 易考点

**1. IAM Role 凭证是临时的（Temporary Credentials）**
Role 与 User 的核心区别：Role 通过 STS（Security Token Service）颁发有时间限制的临时凭证，自动轮换，安全性远高于长期 Access Key。EC2 上的应用应通过 Instance Profile 扮演 Role，而非存储 Access Key。

**2. Instance Store 不支持 Stop 操作，且数据随实例生命周期消失**
Instance Store 是实例存储（本地临时存储），实例终止（Terminate）、失败或被迁移后，数据**永久丢失**，且无法 Stop 后重启。EBS 是现代 EC2 的标准选择，支持 Stop/Start，数据持久。

**3. AMI 是区域资源（Regional），跨区域需显式复制**
AMI 在 us-east-1 创建的只能在 us-east-1 启动实例，要在 eu-west-1 使用，必须通过 CopyImage API 显式复制，且复制不传递权限和标签。

**4. Deregister AMI 不删除底层 EBS Snapshot**
注销 AMI 只是注销模板，对应的 EBS 快照仍然存在并**持续计费**。必须手动删除快照，否则产生"幽灵费用"。

**5. Cloud-Init User Data 默认只在首次启动执行一次**
不是每次重启都执行！如果需要在每次重启时执行，需要使用 `runcmd` 结合 `/etc/rc.local`，或改用 `bootcmd`（每次启动都执行，比 runcmd 更早）。调试时用 `cloud-init clean` + reboot 强制重新运行。

## 自测题

### 题目一（多选题）

以下关于 AWS IAM 的说法，哪些是**正确的**？

A. IAM 是一个区域级（Regional）服务，需要在每个 AWS Region 分别配置  
B. IAM Role 使用临时凭证（Temporary Credentials），比 IAM User 的长期 Access Key 更安全  
C. 应该将 AWS Access Key ID 和 Secret Access Key 存储在 EC2 实例的环境变量中，以便应用程序访问其他 AWS 服务  
D. IAM 策略（Policy）通过 Effect、Action、Resource 三个字段定义允许或拒绝对特定资源的操作  
E. 在云 IAM 中，默认情况下用户对所有 AWS 资源都有访问权限，需要显式拒绝（Deny）才能限制访问  

<details>
<summary>查看答案与解析</summary>

**正确答案：B、D**

- **A ❌**：错误。AWS IAM 是**全局服务（Global Service）**，不与任何特定 Region 关联。在 IAM 中创建的用户、角色、策略在所有 Region 中生效。（对比：EC2、S3、RDS 是区域性服务）

- **B ✅**：IAM Role 通过 AWS STS（Security Token Service）颁发的临时凭证（通常有效期 1–12 小时），自动轮换，不存在泄露后永久有效的风险。IAM User 的 Access Key 是长期凭证，若泄露则需要手动撤销。

- **C ❌**：严重错误！这是最危险的 IAM 反模式之一：
  - Access Key 一旦在环境变量或代码中泄露，任何人都可以使用
  - 正确做法：为 EC2 创建 **IAM Role**，通过 **Instance Profile** 关联，应用程序通过 Instance Metadata Service（IMDS）自动获取临时凭证，无需任何 Access Key

- **D ✅**：IAM Policy 的核心三要素：
  - **Effect**：`Allow` 或 `Deny`
  - **Action**：允许/拒绝的操作（如 `s3:GetObject`、`ec2:DescribeInstances`）
  - **Resource**：操作作用的资源（ARN 格式，如 `arn:aws:s3:::my-bucket/*`）

- **E ❌**：完全相反。AWS IAM 是**默认拒绝（Deny by Default / Secure by Default）**——新用户对任何 AWS 资源都没有访问权限，所有权限必须通过 Policy 显式授予（Allow）。

</details>

### 题目二（多选题）

关于 EC2 机器镜像（AMI）和虚拟化，以下哪些说法是**正确的**？

A. HVM（Hardware Virtual Machine）AMI 利用 CPU 硬件虚拟化扩展，是所有当前代 EC2 实例类型的要求  
B. Instance Store-backed EC2 实例可以被停止（Stop）后重新启动，数据会保留在实例存储中  
C. 当注销（Deregister）一个 EBS-backed AMI 时，对应的 EBS 快照（Snapshot）会被自动一并删除  
D. AMI 是区域资源，将 us-east-1 的 AMI 复制到 eu-west-1 时，原始 AMI 的标签和启动权限不会自动传递  
E. 在黄金镜像（Golden Image）策略中，应用程序的数据库密码应该烘焙（Bake）进镜像，以加快实例启动速度  

<details>
<summary>查看答案与解析</summary>

**正确答案：A、D**

- **A ✅**：AWS 建议所有新实例使用 HVM AMI。HVM（硬件虚拟机）利用 Intel VT-x 或 AMD-V 等 CPU 硬件虚拟化扩展，提供接近原生的性能，并支持增强网络（Enhanced Networking）和 GPU 处理。PV（Paravirtual）是旧版方式，当前代实例类型不支持。

- **B ❌**：错误。Instance Store-backed 实例**不支持 Stop 操作**——实例只能处于 Running 或 Terminated 状态，无法 Stop。而且 Instance Store 是临时存储（Ephemeral），实例一旦终止（Terminate）或发生故障，存储上的数据**永久丢失**。

- **C ❌**：错误。这是一个经典陷阱题！Deregister AMI 只是注销 AMI 模板，**底层的 EBS 快照不会自动删除**，会继续存在并**持续产生存储费用**。需要手动找出并删除这些孤立的快照（Orphaned Snapshots）。

- **D ✅**：AMI 是区域资源（Regional Resource），跨区域复制时（CopyImage API），新 AMI 是独立的副本，**不继承**原始 AMI 的：标签（Tags）、启动权限（Launch Permissions）、S3 存储桶权限。这些必须在目标区域的新 AMI 上重新设置。

- **E ❌**：严重错误！数据库密码等机密信息**绝不能烘焙进镜像**，原因：
  - AMI 可能被共享（公开或与其他账户）
  - AMI 长期存储，密码泄露后难以撤销
  - 不同环境（dev/staging/prod）使用不同密码，无法用同一镜像
  - 正确做法：密码存入 **AWS Secrets Manager**，通过 Cloud-Init 在实例启动时动态拉取

</details>

### 题目三（判断题）

**陈述**：Cloud-Init 的 User Data 配置会在 EC2 实例每次重启时自动重新执行，因此非常适合用来做实例的定期配置更新。

**True or False？**

<details>
<summary>查看答案与解析</summary>

**答案：False（错误）**

**解析：**

Cloud-Init User Data **默认只在实例的首次启动（First Boot）时执行一次**，之后的重启不会重新执行。

**技术原因**：
Cloud-Init 在首次运行后会在 `/var/lib/cloud/` 目录下留下运行记录（semaphore files），再次启动时检测到已执行过则跳过。

**验证方法**：
```bash
ls /var/lib/cloud/instances/<instance-id>/
# 若存在 boot-finished 等标记文件，说明已执行过
```

**如果确实需要每次重启都执行**：

| 方法 | 说明 |
|------|------|
| 使用 `bootcmd` | 比 `runcmd` 更早，**每次启动都执行**（风险：运行多次需保证幂等性） |
| 使用 `/etc/rc.local` 或 systemd 服务 | 传统方式，完全控制执行时机 |
| 强制重新运行 Cloud-Init（测试用）| `sudo cloud-init clean && sudo reboot` |

**云原生正确实践**：
- 定期配置更新 → 使用 **AWS Systems Manager（SSM）**、Ansible 等配置管理工具
- 需要更新配置 → 构建新 AMI，滚动替换实例（不可变基础设施原则）
- Cloud-Init 的定位：**首次启动初始化**，而非持续配置管理

</details>

### 题目四（判断题）

**陈述**：Type 2 Hypervisor（如 VirtualBox）因为运行在宿主操作系统之上，性能开销更大，因此 AWS、GCP、Azure 等主流云提供商使用 Type 2 Hypervisor 来运行用户的虚拟机。

**True or False？**

<details>
<summary>查看答案与解析</summary>

**答案：False（错误）**

**解析：**

说法完全相反。所有主流云提供商都使用 **Type 1（裸金属/Bare-Metal）Hypervisor**，而非 Type 2。

| 对比项 | Type 1（裸金属）| Type 2（托管型）|
|--------|----------------|----------------|
| 运行位置 | **直接在物理硬件上** | 运行在宿主 OS 之上 |
| 性能 | **更高**（无宿主 OS 层开销）| 较低（需穿透宿主 OS）|
| 云提供商使用 | **✅ AWS、GCP、Azure 都用** | ❌ 不用于生产云环境 |
| 典型产品 | **KVM**（AWS/GCP）、**Hyper-V**（Azure）、Xen、VMware ESXi | VirtualBox、Parallels、VMware Workstation |

**各大云提供商的 Hypervisor**：
- **AWS**：最初使用 Xen，现在主要使用基于 **KVM 的 Nitro Hypervisor**（自研，极薄，大量功能卸载到专用硬件）
- **GCP**：使用 **KVM**
- **Azure**：使用 **Hyper-V**（Type 1）

**Type 2 的使用场景**：本地开发、测试、教学——Mac 用 Parallels 跑 Windows、PC 用 VirtualBox 做实验，都是 Type 2 场景。

</details>

### 题目五（场景分析题）

**场景**：

DataSpark 是一家数据分析公司，工程师 Priya 正在为两个不同的分析工作负载选择 AWS EC2 存储方案：

**工作负载 A — 实时流数据处理管道**：
- 使用 Apache Spark 处理来自 Kafka 的实时日志流
- 每批次处理大量临时中间数据（Shuffle 数据），这些数据在作业完成后不需要保留
- 对读写延迟极度敏感，需要最高的本地 I/O 性能
- 作业失败时会从 Kafka 重新消费数据，不需要恢复本地状态

**工作负载 B — 机器学习模型训练与存储**：
- 训练大型深度学习模型，训练过程可能持续数天
- 训练数据集（TB 级别）和模型检查点（Checkpoint）需要在训练中断后能够恢复
- 需要能够暂停实例（节省 GPU 费用），稍后继续训练
- 模型训练完成后需要将模型文件供其他团队使用

请为 Priya 分别推荐工作负载 A 和工作负载 B 的存储方案，并详细说明技术选型理由。

<details>
<summary>查看参考答案</summary>

**参考答案：**

---

**工作负载 A：选择 Instance Store（实例存储）**

**推荐方案**：存储优化型实例（如 `i3` 或 `i4i` 系列）自带的 **Instance Store NVMe SSD**

**技术理由**：

1. **性能优先**：Instance Store 是直接挂载在物理主机上的本地 NVMe SSD，延迟极低（微秒级），吞吐量可达数 GB/s。Spark Shuffle 操作是 I/O 密集型工作，本地磁盘性能直接决定作业速度。

2. **数据是临时的**：Spark 的 Shuffle 中间数据在作业完成后本就需要清理，实例存储的"临时性"不仅不是缺点，反而与业务语义完美匹配——不需要持久化就不需要付费存储。

3. **故障容忍已内建**：Spark + Kafka 架构天然支持故障恢复：
   - Kafka 保存原始消息（有 Retention 策略）
   - Spark 作业失败后可以从 Kafka Offset 重新消费，重建中间数据
   - 因此 Instance Store 的"数据随实例消失"不会影响业务正确性

4. **成本合理**：Instance Store 的费用包含在实例价格中，不需要额外付 EBS 的 IOPS/GB 费用。对于高 I/O 的 Spark 作业，这通常比同等 EBS 配置便宜。

**注意事项**：
- Instance Store-backed 实例**不支持 Stop**，只能 Terminate。如果要暂停节省费用，需要将任何需要保留的状态（如 Spark 检查点）写入 **S3**，而非实例存储

---

**工作负载 B：选择 EBS（弹性块存储）**

**推荐方案**：GPU 实例（如 `p3` 或 `p4d`）+ **EBS gp3 或 io2 卷**

**技术理由**：

1. **必须支持 Stop/Start**：训练持续数天，需要在低峰期或不用时停止实例（释放 GPU 费用）后继续。Instance Store 不支持 Stop，实例一停数据全丢，不可接受。EBS 卷独立于实例生命周期，Stop → Start 后数据完整保留。

2. **模型检查点需要持久化**：深度学习训练每隔若干步保存检查点（Checkpoint），允许从中断处恢复。这些检查点必须持久存储，EBS 是最直接的选择：写入 EBS 就等于持久化。

3. **跨团队共享模型文件**：训练完成的模型文件可以：
   - 保存在 EBS 上，创建 EBS 快照（Snapshot），共享给其他团队
   - 或从 EBS 上传到 **S3**，通过 S3 URL 供其他团队下载

4. **数据集管理**：TB 级训练数据集建议存放在 **S3**（成本低、持久），通过 Instance Store 或 EBS 作为本地缓存加速读取。

**推荐 EBS 卷类型**：
| 场景 | 推荐卷类型 | 理由 |
|------|-----------|------|
| 根卷（OS）| gp3 | 通用，低成本 |
| 训练数据缓存 | gp3（高吞吐配置）| 顺序读写，高吞吐优先 |
| 模型检查点 | gp3 或 io2 | 需要可靠的写入性能 |

---

**总结对比**

| 维度 | 工作负载 A（流处理）| 工作负载 B（模型训练）|
|------|-------------------|---------------------|
| 存储方案 | **Instance Store** | **EBS** |
| 核心需求 | 极高 I/O、临时数据 | 持久化、支持 Stop/Start |
| 数据失效影响 | 可接受（Kafka 重消费）| 不可接受（训练进度丢失）|
| 实例 Stop 需求 | 不需要 | **必须**（节省 GPU 费用）|
| 关键决策因素 | 性能 + 数据临时性 | 持久性 + 实例可停止性 |

**Priya 的总结原则**：
> 当数据是**临时中间结果**且对 **I/O 性能有极高要求**时，选 Instance Store；当数据需要**持久化**、实例需要**Stop 后恢复**，或需要**跨实例/跨账户共享**时，选 EBS。

</details>
