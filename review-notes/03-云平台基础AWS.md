# L01 + L03 — 云计算基础与 AWS 全球基础设施

## 模块总览

本模块建立云计算的理论基石：用 **NIST 五大特征**判断"什么是真正的云"，理解 IaaS/PaaS/SaaS 三种服务模型的责任边界，再落地到 AWS 全球基础设施的三级结构（Region → AZ → Edge Location）。

---

## 核心概念

### 一、NIST 云计算定义（必考！）

**NIST（美国国家标准与技术研究院）官方定义**：
云计算是一种模型，可以按需、方便地通过网络访问可配置的计算资源共享池（网络、服务器、存储、应用和服务），这些资源能够被快速供应和释放，且所需管理工作或服务提供商交互极少。

NIST 定义包含三个维度：
- **5 个基本特征（Essential Characteristics）**
- **3 种服务模型（Service Models）**
- **4 种部署模型（Deployment Models）**

---

### 二、五大基本特征（Essential Characteristics）

| 特征 | 英文 | 核心含义 | 举例 |
|------|------|----------|------|
| **按需自助服务** | On-Demand Self-Service | 用户无需联系服务提供商，自行配置计算资源 | 在 AWS 控制台点击创建 EC2 实例，无需打电话给 AWS |
| **广泛网络访问** | Broad Network Access | 通过标准网络机制访问，支持多种客户端（手机、笔记本、服务器） | 从任何地方用浏览器访问 AWS Console |
| **资源池化** | Resource Pooling | 多租户（Multi-Tenancy）共享物理资源，用户不知道/不控制资源的确切物理位置 | 多个客户共享同一台物理服务器的虚拟化资源 |
| **快速弹性** | Rapid Elasticity | 资源可以快速、弹性地扩展或收缩，对用户来说看起来像无限 | Auto Scaling 在流量峰值时自动增加实例，峰值后自动缩减 |
| **计量服务** | Measured Service | 资源使用被监控、控制、报告，按实际使用量计费（Pay-as-You-Go） | AWS 按小时/GB/请求次数精确计费 |

> ⭐ **记忆口诀（首字母）**：**O**n-demand, **B**road, **R**esource Pooling, **R**apid Elasticity, **M**easured = **OBRRM**

### 三、三种服务模型（Service Models）

#### 责任边界对比（"你管谁"）

```
责任                     IaaS        PaaS        SaaS
────────────────────────────────────────────────────
Application（应用）       你           你          提供商
Data（数据）              你           你          提供商
Runtime（运行时）         你           提供商      提供商
Middleware（中间件）      你           提供商      提供商
Operating System（操作系统）你         提供商      提供商
────────────────────────────────────────────────────
Virtualization（虚拟化）  提供商       提供商      提供商
Servers（服务器）         提供商       提供商      提供商
Storage（存储）           提供商       提供商      提供商
Networking（网络）        提供商       提供商      提供商
```

| 服务模型 | 定义 | AWS 举例 | 典型用户 |
|----------|------|----------|----------|
| **IaaS**（基础设施即服务） | 提供虚拟化计算、存储、网络资源；你管操作系统及以上 | EC2, S3, VPC | 系统管理员、DevOps 工程师 |
| **PaaS**（平台即服务） | 提供应用运行平台；你只管应用代码和数据 | Elastic Beanstalk, AWS Lambda, RDS | 应用开发者 |
| **SaaS**（软件即服务） | 提供完整应用；你只是用它 | Gmail, Salesforce, Office 365 | 最终用户 |

> **FaaS（函数即服务，Function as a Service）**：比 PaaS 更细粒度，AWS Lambda 是典型代表，按函数调用次数计费，无需关心服务器。

### 四、四种部署模型（Deployment Models）

| 部署模型 | 定义 | 典型场景 | 举例 |
|----------|------|----------|------|
| **公有云（Public Cloud）** | 云基础设施由第三方提供商拥有、运营，通过互联网向公众提供 | 初创公司、Web 应用 | AWS、GCP、Azure |
| **社区云（Community Cloud）** | 由具有共同关切（使命、安全要求、合规）的特定组织社区共享 | 政府机构、医疗行业 | **AWS GovCloud**（仅面向美国政府）|
| **私有云（Private Cloud）** | 云基础设施专为单一组织运营，可由组织或第三方管理 | 金融、医疗等合规行业 | VMware Cloud、OpenStack |
| **混合云（Hybrid Cloud）** | 两种或多种部署模型组合，通过标准技术绑定在一起 | 敏感数据私有，高弹性需求公有 | 本地数据中心 + AWS Direct Connect |

> ⭐ **关键易混点：私有云（Private Cloud）≠ 本地部署（On-Premise）**
> - 私有云满足 NIST 五大特征（弹性、按需自助、计量服务）
> - 本地部署通常是固定容量，不满足云计算特征
> - 区别：私有云仍然是"云"，本地部署不是"云"

### 五、AWS 全球基础设施（Global Infrastructure）

AWS 全球基础设施分为三个层级，从大到小：

#### 层级 1：区域（Region）
**定义**：特定地理位置的资源托管区域，是可用区（AZ）的集合。

**关键特性**：
- 部署在一个 Region 的资源**不会自动复制**到其他 Region
- **数据主权（Data Sovereignty）**：某些数据依法必须留在特定地理边界内（如 GDPR 要求 EU 公民数据留在欧盟）
- Region 之间完全隔离，各自拥有独立基础设施

**Region 选择因素**：合规要求 → 延迟 → 成本 → 可用服务

举例：`us-east-1`（弗吉尼亚）、`eu-west-1`（爱尔兰）、`ap-south-1`（孟买）

#### 层级 2：可用区（Availability Zone, AZ）
**定义**：Region 内的部署区域，是**逻辑隔离边界（Logical Isolation Boundary）**。

**关键特性（必考！）**：
- **AZ ≠ 单个数据中心**：一个 AZ 可能由**多个数据中心**组成
- 每个数据中心有独立的：电源供应、冷却系统、物理安全设施
- AZ 之间通过**低延迟、高吞吐、高冗余**网络连接
- 典型区内延迟（Inter-AZ Latency）：**1–2 毫秒**，支持**同步复制（Synchronous Replication）**
- 将 AZ 理解为"故障域（Failure Domain）"：一个 AZ 故障，其他 AZ 继续运行

```
AWS Region（例：us-east-1）
├── AZ-1a（us-east-1a）
│   ├── 数据中心 A
│   └── 数据中心 B
├── AZ-1b（us-east-1b）
│   └── 数据中心 C
└── AZ-1c（us-east-1c）
    └── 数据中心 D
```

#### 层级 3：边缘节点（Edge Location）
**定义**：分布在全球各大城市的**内容分发网络（CDN）接入点**，比 Region 数量多得多（400+ 个），离最终用户更近。

**作用**：
- **CloudFront（CDN）**：缓存静态内容（图片、视频、网页）在最近的 Edge Location，降低延迟
- **Route 53（DNS）**：DNS 查询在边缘节点响应，加速解析
- **AWS Global Accelerator**：将用户流量路由到最近的 AWS 接入点
- **AWS WAF / Shield**：在边缘节点过滤恶意流量（DDoS 防护）

**关键对比**：
```
Region（30+ 个）：全量 AWS 服务，有 AZ，可运行计算/数据库
Edge Location（400+ 个）：仅用于内容缓存和流量加速，不运行 EC2 等计算
```

### 六、云计算的业务驱动因素

| 驱动因素 | 说明 | 核心转变 |
|----------|------|----------|
| **容量规划（Capacity Planning）** | 传统本地部署：平均服务器利用率仅 **20%** 左右；提前规划浪费大量资源 | 按需弹性替代提前规划 |
| **成本降低（Cost Reduction）** | 资本支出（CapEx）→ 运营支出（OpEx）；从买服务器变成按用量付费 | CapEx → OpEx |
| **组织敏捷性（Organizational Agility）** | 从需求提出到资源就绪：传统数周/数月 → 云端几分钟 | 分钟级速度替代周/月级 |

**容量规划策略**：
| 策略 | 说明 | 风险 |
|------|------|------|
| Lead（超前）| 提前扩容，保证性能 | 资源浪费 |
| Lag（滞后）| 需求已出现才扩容 | 服务不可用风险 |
| Match（匹配）| 小步增量跟进需求 | 最平衡但执行复杂 |

## 概念间关系

```
NIST 云计算定义
    ├── 5大特征（判断"是否是云"的标准）
    │       ↓ 在 AWS 中如何体现？
    │   On-demand Self-Service → AWS Console / API 自助创建资源
    │   Broad Network Access   → 全球通过互联网访问 AWS
    │   Resource Pooling       → 多租户虚拟化（EC2 共享物理机）
    │   Rapid Elasticity       → Auto Scaling / Lambda 按需伸缩
    │   Measured Service       → AWS Cost Explorer / Pay-as-you-go
    │
    ├── 3种服务模型（责任边界划分）
    │   IaaS（EC2）→ PaaS（Beanstalk）→ SaaS（外部应用）
    │   管理责任逐步从用户转移到提供商
    │
    └── 4种部署模型（隐私/合规/成本权衡）
            Public（最灵活）→ Hybrid（平衡）→ Private（最合规）

AWS 全球基础设施
    Region（地理位置，数据主权边界）
        └── AZ（故障隔离域，同步复制单元，1-2ms 延迟）
                └── Data Center（物理设施）
    Edge Location（CDN 缓存节点，用户就近访问）

高可用设计原则：
    Multi-AZ → 应对 AZ 级故障（最常见）
    Multi-Region → 应对区域级灾难（配合 Edge Location 全球加速）
```

## 对比表格

### 1. IaaS vs PaaS vs SaaS
| 维度 | IaaS | PaaS | SaaS |
|------|------|------|------|
| 用户管理内容 | OS + Runtime + App + Data | App + Data | 无（仅使用） |
| 控制粒度 | 最高 | 中等 | 最低 |
| 所需技能 | 系统管理、网络 | 应用开发 | 无需技术 |
| 灵活性 | 最高 | 中等 | 最低 |
| 启动速度 | 慢（需配置 OS） | 快 | 立即使用 |
| AWS 示例 | EC2, S3, VPC | Elastic Beanstalk, Lambda, RDS | 无（AWS 不提供 SaaS） |

### 2. 四种部署模型对比
| 模型 | 谁拥有基础设施 | 谁能访问 | 合规优势 | 成本 |
|------|--------------|----------|----------|------|
| Public Cloud | 第三方（AWS等） | 任何人（共享） | 低（默认共享） | 最低（按需付费） |
| Community Cloud | 共享（特定社区） | 特定行业/机构 | 高（行业定制） | 中等 |
| Private Cloud | 单一组织 | 仅本组织 | 最高 | 高 |
| Hybrid Cloud | 混合 | 部分公开/部分私有 | 可定制 | 中等 |

### 3. Region vs AZ vs Edge Location
| 维度 | Region | Availability Zone | Edge Location |
|------|--------|-------------------|---------------|
| 定义 | 地理区域 | 区域内逻辑隔离单元 | CDN 接入点 |
| 数量（约） | 30+ 个 | 90+ 个 | 400+ 个 |
| 等同于 | 省/州级地理位置 | 一组数据中心 | 城市级缓存节点 |
| 主要用途 | 运行完整 AWS 服务 | 高可用部署单元 | 内容缓存/DNS 加速 |

| 跨单元延迟 | 数十–数百 ms | **1–2 ms** | 最低（直连用户） |

### 4. 水平扩展 vs 垂直扩展
| 维度 | 水平扩展（Scale Out/In） | 垂直扩展（Scale Up/Down） |
|------|------------------------|--------------------------|
| 方式 | 增加/减少实例数量 | 升级/降级单实例规格（CPU、内存） |
| 对应 NIST 特征 | Rapid Elasticity | Rapid Elasticity（有限） |
| 停机需要？ | ❌ 不需要 | ✅ 通常需要重启 |
| 弹性上限 | 理论上无限 | 受单机最大规格限制 |
| 成本 | 按实例数量线性增加 | 高配置实例单价急剧上升 |
| 云原生首选 | ✅ 是 | 适合短期快速调整 |

### 5. 私有云 vs 本地部署（易混淆）
| 维度 | 私有云（Private Cloud） | 本地部署（On-Premise） |
|------|----------------------|----------------------|
| 满足 NIST 五大特征？ | ✅ 是 | ❌ 否 |
| 弹性 | ✅ 有（软件定义） | ❌ 固定容量 |
| 按需自助服务 | ✅ 有 | ❌ 无 |
| 本质 | 是"云"，只是专属 | 不是云 |
| 举例 | VMware vSphere + 云管平台 | 机房里的物理服务器 |

## ⭐ 易考点

**1. NIST 五大特征逐一辨认**
考试高频：给出一段业务描述，要求判断是否满足某个特征。重点区分：
- **资源池化（Resource Pooling）**：关键词是"多租户（Multi-Tenancy）"，用户不控制资源的确切物理位置
- **计量服务（Measured Service）**：关键词是"按用量付费"、"监控资源使用"
- **快速弹性（Rapid Elasticity）**：关键词是"自动扩缩容"、"看起来像无限资源"

**2. AZ ≠ 数据中心（高频考点）**
一个 AZ 由**多个物理数据中心**组成，每个数据中心有独立电源、冷却和物理安全。AZ 是逻辑隔离边界，不是单栋建筑。

**3. 区内延迟 1-2ms 的意义**
这个数字的重要性在于：它使**同步复制**成为可能。跨区域延迟数百 ms，只能做异步复制。Multi-AZ 高可用的底层依赖就是这个低延迟。

**4. 私有云≠本地部署**
私有云仍然是"云"（满足 NIST 五大特征），本地部署不是。区分依据：私有云有弹性、有按需自助、有计量服务；本地部署是固定容量、手工运维。

**5. CapEx vs OpEx 转变**
IaaS/Cloud 的核心商业价值：将一次性资本支出（服务器购置）转变为可变运营支出（按用量付费）。传统本地服务器平均利用率仅 **20%**，80% 的投资被浪费。

### 题目一（多选题）

根据 NIST 云计算定义，以下哪些特征属于云计算的**五大基本特征（Essential Characteristics）**？

A. 按需自助服务（On-Demand Self-Service）：用户无需联系服务提供商即可单方面配置计算资源  
B. 低成本（Low Cost）：云计算必须比本地部署便宜  
C. 资源池化（Resource Pooling）：提供商的计算资源被汇集起来，通过多租户模型服务于多个消费者  
D. 快速弹性（Rapid Elasticity）：资源可以弹性地扩展和释放，对消费者来说看起来像无限  
E. 计量服务（Measured Service）：云系统自动控制和优化资源使用，使用量可被监控、控制、报告  

<details>
<summary>查看答案与解析</summary>

**正确答案：A、C、D、E**

- **A ✅**：按需自助服务（On-Demand Self-Service）是 NIST 第一个特征。关键词：**用户无需联系服务提供商**，单方面自行配置。例如：在 AWS Console 点击创建 EC2，无需打电话或提交工单给 AWS。
- **B ❌**：**低成本不是 NIST 五大特征之一**。云计算在某些场景下甚至比本地部署更贵（如持续运行的大型工作负载）。NIST 定义关注的是**技术特征**，不是价格。
- **C ✅**：资源池化（Resource Pooling）。关键词：**多租户（Multi-Tenancy）**，用户通常不知道或不控制资源的确切物理位置。
- **D ✅**：快速弹性（Rapid Elasticity）。关键词：**弹性扩缩**，对用户来说资源**看起来像无限**（Appears to be unlimited）。
- **E ✅**：计量服务（Measured Service）。关键词：**自动监控、控制、报告**资源使用，实现**按用量付费（Pay-as-you-go）**。

**NIST 五大特征完整列表**：
1. On-Demand Self-Service（按需自助）
2. Broad Network Access（广泛网络访问）
3. Resource Pooling（资源池化）
4. Rapid Elasticity（快速弹性）
5. Measured Service（计量服务）

</details>

### 题目二（多选题）

关于 AWS 全球基础设施的以下陈述，哪些是**正确的**？

A. 每个 AWS 可用区（AZ）恰好对应一个物理数据中心  
B. 部署在 us-east-1（弗吉尼亚）Region 的 S3 存储桶中的数据，会自动复制到 eu-west-1（爱尔兰）以提高可用性  
C. AWS Edge Location 的数量远多于 AWS Region 的数量，主要用于 CloudFront CDN 内容缓存  
D. 同一 Region 内不同 AZ 之间的典型延迟约为 1-2 毫秒，足以支持数据库的同步复制  
E. 选择 AWS Region 时需要考虑数据主权法规（如 GDPR），因为数据可能被要求不能离开特定地理边界  

<details>
<summary>查看答案与解析</summary>

**正确答案：C、D、E**

- **A ❌**：错误。这是最常见的误解！一个 AZ **可以包含多个物理数据中心**，每个数据中心有独立的电源、冷却和物理安全。AZ 是**逻辑隔离边界**，不是单栋建筑。AWS 官方定义：AZ 由"一个或多个独立数据中心"组成。
- **B ❌**：错误。**不同 Region 之间没有自动数据复制**。跨 Region 复制需要用户显式配置（如 S3 跨区域复制 CRR）。AWS Region 之间完全隔离，各自独立。
- **C ✅**：Edge Location（400+）远多于 Region（30+）。Edge Location 主要支持 CloudFront CDN、Route 53 DNS、AWS Global Accelerator，将内容缓存在离用户最近的节点。
- **D ✅**：1-2 ms 是 AZ 间的关键技术规格。这个低延迟使**同步复制**（写操作同时确认到多个 AZ）成为可能，是 Multi-AZ 高可用架构的基础。跨 Region 延迟数百 ms，只能做异步复制。
- **E ✅**：数据主权（Data Sovereignty）是 Region 选择的重要合规因素。GDPR 要求欧盟公民数据保留在欧盟（选 EU Region）；部分国家有数据本地化要求（如中国、俄罗斯）。

</details>

### 题目三（判断题）

**陈述**：AWS GovCloud（专为美国政府打造的 AWS 区域）属于"私有云（Private Cloud）"部署模型，因为它不对普通公众开放。

**True or False？**

<details>
<summary>查看答案与解析</summary>

**答案：False（错误）**

**解析：**

AWS GovCloud **不是**私有云（Private Cloud），而是**社区云（Community Cloud）**。

根据 NIST 四种部署模型的定义：

| 模型 | 核心特征 |
|------|---------|
| 私有云（Private Cloud） | 基础设施**专为单一组织**（one organization）运营 |
| 社区云（Community Cloud） | 基础设施由**具有共同关切的特定社区**共享使用（specific community of consumers from organizations that have shared concerns） |

**AWS GovCloud 的关键特征**：
- 服务对象：**多个**美国政府机构、承包商、合规要求实体
- 不是某一个政府机构专用的——它是**多个机构共享**的基础设施
- 共同关切：合规要求（FedRAMP、ITAR）、安全标准、数据主权（数据必须在美国）

**因此**：GovCloud = **社区云**（特定社区 = 美国政府相关机构）。

**类似例子**：
- 医疗行业专用云（符合 HIPAA）→ 社区云
- 金融行业合规云（符合 PCI-DSS）→ 社区云
- 某银行自建专用数据中心 + 云平台 → 私有云

**关键判断标准**：服务单一组织 → 私有云；服务有共同需求的多个组织 → 社区云。

</details>

### 题目四（判断题）

**陈述**：一家公司将所有应用迁移到自己机房内部部署的服务器上（On-Premise），并声称已经实现了"私有云（Private Cloud）"。这种说法是正确的。

**True or False？**

<details>
<summary>查看答案与解析</summary>

**答案：False（错误）**

**解析：**

这是 Cloud Computing 中**最常见的混淆之一**：私有云 ≠ 本地部署。

**判断标准：是否满足 NIST 五大特征**

| NIST 特征 | 私有云 | 典型本地部署（服务器机房） |
|-----------|--------|--------------------------|
| 按需自助服务 | ✅ 有（云管平台自助申请） | ❌ 通常需要提交工单/等待 IT |
| 广泛网络访问 | ✅ 有 | ✅ 有（内网或VPN） |
| 资源池化（多租户） | ✅ 虚拟化共享资源 | ❌ 通常物理机固定分配 |
| 快速弹性 | ✅ 有（软件定义扩缩容） | ❌ 固定容量，扩容需采购 |
| 计量服务 | ✅ 有（内部计量收费） | ❌ 通常无精细计量 |

**结论**：
- 该公司的做法是**本地部署（On-Premise）**，不满足 NIST 的多项特征（尤其是弹性、按需自助、计量服务）
- **私有云的本质是"云"**——只是该云的资源专属于一个组织，但仍然满足所有云计算特征
- **本地部署的本质是"非云"**——是传统 IT 基础设施模式

**正确的私有云示例**：
- 使用 VMware vSphere + OpenStack 构建内部资源池，员工通过自助门户申请虚拟机，IT 部门按使用量向各部门分摊费用 → 这才是私有云

</details>

### 题目五（场景分析题）

**场景**：

公司技术总监王总在全员大会上宣布："我们已经完成云化转型！我们在公司机房里部署了 200 台高性能服务器，每个业务团队分配固定数量的服务器，直接远程登录使用，所有服务器都通过公司内网访问。我们现在既省了 AWS 的费用，又保证了数据安全，这就是我们自己的私有云！"

作为公司的云计算工程师 Kevin，你需要用 **NIST 五大特征**来评估：王总宣布的方案**是否真的是云计算（私有云）**？

请逐一分析五大特征的满足情况，并给出你的最终判断和建议。

<details>
<summary>查看参考答案</summary>

**参考答案：**

**整体结论：王总的方案不是云计算，而是传统本地部署（On-Premise）。**

---

**逐一分析 NIST 五大特征：**

**① 按需自助服务（On-Demand Self-Service）**
> *"每个业务团队分配固定数量的服务器，直接远程登录使用"*

**❌ 不满足**

真正的云需要用户**无需联系服务提供商（IT 部门）**，自行按需申请资源。该方案中：
- 服务器是**预先分配的固定数量**，不是按需申请
- 如果某业务需要更多资源，需要找 IT 部门协调或采购新硬件
- 没有自助服务门户（Portal）

---

**② 广泛网络访问（Broad Network Access）**
> *"通过公司内网访问"*

**⚠️ 部分满足（有争议）**

内网访问勉强符合"通过网络访问"的要求，但 NIST 要求"通过标准机制，支持异构客户端平台"。仅限内网会限制远程办公和移动设备访问。这条勉强给过，但评分不高。

---

**③ 资源池化（Resource Pooling / Multi-Tenancy）**
> *"每个业务团队分配固定数量的服务器"*

**❌ 不满足**

资源池化的核心是：计算资源被**汇集**成共享池，**动态分配**给多个消费者。该方案中：
- 每个团队有**固定**分配的服务器，物理隔离
- 资源不能在团队间动态调配（A 团队服务器空闲时，B 团队忙碌时无法借用）
- 没有虚拟化实现多租户共享

---

**④ 快速弹性（Rapid Elasticity）**
> *"200 台高性能服务器"（固定数量）*

**❌ 不满足**

弹性要求资源能**快速扩展和收缩**，对用户来说像无限资源。该方案中：
- 200 台服务器是**固定上限**
- 如果某团队突发流量需要更多资源：要等 IT 采购（数周/数月）
- 没有任何弹性伸缩机制

这正是传统本地部署最大的问题：**过度配置浪费（服务器平均利用率约 20%）**或**配置不足影响业务**。

---

**⑤ 计量服务（Measured Service）**

**❌ 不满足（方案中未提及）**

真正的云需要自动**监控、控制、报告**资源使用量，实现按需付费或内部分摊。该方案中没有提到任何计量或资源使用追踪机制。

---

**最终评估表：**

| NIST 特征 | 满足？ | 关键缺陷 |
|-----------|--------|---------|
| 按需自助服务 | ❌ 否 | 固定分配，无自助申请 |
| 广泛网络访问 | ⚠️ 部分 | 仅限内网 |
| 资源池化 | ❌ 否 | 物理固定分配，无动态共享 |
| 快速弹性 | ❌ 否 | 固定 200 台，无弹性 |
| 计量服务 | ❌ 否 | 无监控和计量 |

**判断：王总的方案是传统本地部署（On-Premise），不是私有云。**

---

**Kevin 给王总的建议：**

如果公司因数据安全要求必须使用自有基础设施，可以考虑真正的**私有云方案**：

1. **增加虚拟化层**：用 VMware/OpenStack 将 200 台物理服务器变成虚拟资源池
2. **搭建自助服务门户**：员工自行申请虚拟机/容器，IT 部门审批策略而非手动分配
3. **实现资源池动态调配**：资源在团队间按需共享，提高利用率（目标从 20% 提升到 60%+）
4. **建立内部计量系统**：按各团队实际使用量内部结算，激励合理使用资源
5. **弹性扩展路径**：高峰期与公有云（AWS）混合使用（Hybrid Cloud），保持日常数据本地，弹性算力上公有云

</details>
