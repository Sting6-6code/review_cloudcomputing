## 核心概念

### 1. 微服务设计原则 (Microservices Design Principles)

#### 单体 vs 微服务

**单体 (Monolith)**：单一可部署制品，共享数据库、进程和部署生命周期。
- 优点：开发/测试/调试简单，延迟低，事务一致性强
- 缺点：代码膨胀、全量扩展、技术锁定、团队耦合

**微服务 (Microservices)**：每个服务独立部署、独立扩展、拥有自己的数据。

两大支柱：
- **松耦合 (Loose Coupling)**：修改一个服务无需修改另一个
- **高内聚 (High Cohesion)**：相关逻辑集中在同一服务

#### 关键设计决策

| 概念 | 说明 |
|------|------|
| **每服务独立数据库 (Database per Service)** | 不共享数据库，防止依赖泄露 |
| **共享数据库反模式 (Shared DB Anti-Pattern)** | 破坏独立性，最危险的反模式 |
| **API Gateway** | 单一入口，处理路由/认证/限流/监控 |
| **Conway's Law** | 系统架构镜像组织结构；"两块披萨团队" |
| **Saga 模式** | 跨服务分布式事务：编排型 (Orchestration) vs 协调型 (Choreography)；补偿事务 (Compensating Transaction) |
| **最终一致性 (Eventual Consistency)** | 无跨服务 ACID，服务间只能最终一致 |
| **Strangler Fig 模式** | 渐进式从单体迁移，不做大爆炸重写 |

#### 何时不用微服务
- 团队 < 5-10 人
- 领域边界不清晰
- MVP / 快速验证阶段
- 简单 CRUD 应用

> Martin Fowler：**"不要从微服务开始。"** (Don't start with microservices.)

### 2. 事件驱动架构 (Event-Driven Architecture, EDA)

EDA 减少三种耦合：
- **空间耦合 (Spatial Coupling)**：生产者不需要知道消费者的位置
- **时间耦合 (Temporal Coupling)**：生产者无需消费者同时在线
- **行为耦合 (Behavioral Coupling)**：生产者不关心消费者如何处理

#### 核心组件

| 组件 | 说明 | AWS 对应 |
|------|------|---------|
| 生产者 (Producer) | 发布事件/消息 | 任意服务 |
| 消费者 (Consumer) | 订阅并处理 | Lambda / EC2 |
| 通道 (Channel) | Topic / Queue | SNS Topic / SQS Queue |
| 代理 (Broker) | 管理消息路由 | SNS / SQS / EventBridge |

#### SNS vs SQS 核心差异

| 特性 | SNS (Simple Notification Service) | SQS (Simple Queue Service) |
|------|----------------------------------|---------------------------|
| 模式 | Pub/Sub 扇出 (Fan-out) | 点对点队列 (Point-to-point) |
| 推/拉 | 推送 (Push-based) | 拉取 (Pull-based) |
| 消息保留 | 无持久存储 | 最长 14 天 |
| 消费者 | 所有订阅者都收到副本 | 只有一个消费者收到 |
| 支持协议 | SQS / Lambda / HTTP / Email / SMS | — |
| FIFO 支持 | 有 (精确一次) | 有 (精确一次) |

#### 消息投递保证 (Delivery Guarantees)

| 级别 | 说明 | 适用场景 |
|------|------|---------|
| 至多一次 (At-most-once) | 可能丢消息 | 日志、指标 |
| 至少一次 (At-least-once) | 可能重复，需幂等 | 默认 SQS/SNS |
| 精确一次 (Exactly-once) | FIFO 队列 | 支付、订单 |

#### 关键机制
- **死信队列 (DLQ, Dead Letter Queue)**：超过 `maxReceiveCount` 次失败后转入 DLQ；应监控 DLQ 告警为零消息
- **可见性超时 (Visibility Timeout)**：SQS 默认 30 秒；处理期间消息对其他消费者不可见
- **长轮询 (Long Polling)**：`WaitTimeSeconds` 最长 20 秒，优于短轮询（减少空请求）
- **幂等性 (Idempotency)**：幂等键 (UUID) + 已处理 ID 表；条件写入；天然幂等操作 (SET vs INCREMENT)
- **指数退避 + 抖动 (Exponential Backoff + Jitter)**：避免惊群效应 (Thundering Herd)

#### 消息模式

| 模式 | 说明 |
|------|------|
| 扇出 (Fan-out) | 一 → 多，SNS → 多个 SQS/Lambda |
| 扇入 (Fan-in) | 多 → 一，聚合多来源 |
| 负载均衡 (Load Balancing) | 多生产者 → 多消费者，每消息只被一个处理 |

### 3. Serverless / FaaS

#### 抽象层次演进

```
物理机 → 虚拟机 (抽象硬件) → 容器 (抽象 OS) → Serverless (抽象运行时)
```

#### FaaS 核心特征
- 云厂商管理服务器；事件触发；自动扩缩到零；按毫秒计费；无状态短暂执行

#### Lambda 冷启动 (Cold Start) 流程

```
下载代码包 (~100ms) → 启动运行时 → 执行 INIT 代码（全局变量、DB 连接）→ 执行 handler
```

冷启动影响因素：
- 编程语言（Go/Rust 快；Java/Spring 慢）
- 包体大小
- 内存分配大小
- VPC 配置（增加约 100-200ms）

#### AWS Lambda 重要限制

| 参数 | 限制 |
|------|------|
| 内存 | 128 MB – 10,240 MB |
| 超时 | 最长 15 分钟 (900s) |
| 部署包 (ZIP) | 50 MB 压缩 / 250 MB 解压 |
| 容器镜像 | 10 GB |
| 并发 | 默认 1,000 |
| Layer 数量 | 最多 5 个 |
| 计费 | 前 100 万次请求免费；之后 $0.20/百万次 + 按内存×时长计费 |

#### 适用 vs 不适用

| 适合 Serverless | 不适合 Serverless |
|----------------|-----------------|
| 事件驱动工作负载 | 长时运行任务 |
| 流量波动的 REST API | 持久化连接需求 |
| 定时任务 / ETL | HPC / GPU 计算 |
| MVP 快速迭代 | 高吞吐稳定流量 |
| Spiky 流量 | 极低延迟要求 |

#### 决策框架
1. 需要持续运行？→ VM / Container
2. 能在 15 分钟内完成？→ Serverless (Lambda)
3. 超过 15 分钟？→ Step Functions / Container

### 4. 云安全最佳实践 (Cloud Security Best Practices)

#### CIA 三元组 + 第四支柱

| 支柱 | 含义 | 技术手段 |
|------|------|---------|
| 机密性 (Confidentiality) | 只有授权方能读取 | 加密 (Encryption) |
| 完整性 (Integrity) | 数据未被篡改 | 哈希 / 数字签名 |
| 可用性 (Availability) | 服务持续可用 | DDoS 防护、冗余 |
| 真实性 (Authenticity) | 身份可验证 | 数字证书、签名 |

#### 共担责任模型 (Shared Responsibility Model)

| 服务模型 | 客户负责 |
|---------|---------|
| IaaS | OS 补丁、应用代码、IAM、网络配置、加密、日志 |
| PaaS | 应用代码、数据、访问控制、函数配置 |
| SaaS | 数据、访问控制 |

云厂商负责：**云基础设施安全 (Security OF the cloud)**  
客户负责：**云上资源安全 (Security IN the cloud)**

#### 加密类型对比

| 类型 | 特征 | 用途 |
|------|------|------|
| 对称加密 (Symmetric, AES-256) | 同一密钥加解密，速度快 | 大量数据加密 |
| 非对称加密 (Asymmetric, RSA) | 公私钥对，慢 100-1000x | TLS 握手、数字签名 |
| 哈希 (Hashing, SHA-256) | 单向不可逆，确定性 | 密码存储、完整性校验 |

**TLS 握手**：非对称加密交换对称会话密钥 → 之后用对称密钥加密通信。

#### 静态加密默认状态 (Encryption at Rest Defaults)

| 服务 | 默认加密 |
|------|---------|
| S3 | ✅ SSE-S3 默认开启 |
| GCS | ✅ Google 托管密钥 |
| EBS | ❌ 需账户级别开启 |
| RDS | ❌ 必须在创建时启用 |
| Azure SQL | ✅ TDE 默认开启 |

#### 防火墙层级

| 类型 | 层级 | 有状态 | 规则 |
|------|------|--------|------|
| Security Group (AWS) | 实例/ENI | ✅ 有状态 | 仅允许 (Allow-only) |
| NACL (AWS) | 子网 | ❌ 无状态 | 允许 + 拒绝，有优先级 |
| GCP Firewall Rules | VPC 级 | ✅ 有状态 | 允许 + 拒绝 |
| Azure NSG | 子网或 NIC | ✅ 有状态 | 允许 + 拒绝 |

**WAF (Web Application Firewall)**：检查 L7 HTTP/HTTPS 流量；AWS WAF (ALB/CloudFront/API GW)；防 SQL 注入、XSS 等。

#### OWASP Top 10 (2021)

1. **Broken Access Control** — 访问控制失效
2. **Cryptographic Failures** — 加密失败
3. **Injection** — 注入攻击（SQL 注入）
4. **Insecure Design** — 不安全设计
5. **Security Misconfiguration** — 安全配置错误
6. **Vulnerable & Outdated Components** — 脆弱组件
7. **Identification & Authentication Failures** — 认证失败
8. **Software & Data Integrity Failures** — 完整性失败
9. **Security Logging & Monitoring Failures** — 日志监控不足
10. **Server-Side Request Forgery (SSRF)**

> **SQL 注入防御**：参数化查询 (Parameterized Queries / Prepared Statements)；使用 RDS/Cloud SQL **不能**自动防止 SQLi。

#### 密钥与凭证管理

- **KMS**：AWS KMS / GCP Cloud KMS / Azure Key Vault；硬件安全模块 (HSM) 支持；支持客户托管密钥 (CMK/BYOK)
- **Secrets Manager**：AWS Secrets Manager ($0.40/secret/月)；GCP Secret Manager；Azure Key Vault
- **硬编码凭证 = 灾难**：GitHub 机器人几分钟内就能扫到；Capital One 2019 泄露 → $300M 损失

#### 审计日志

| 服务 | 默认保留 | 备注 |
|------|---------|------|
| AWS CloudTrail | 90 天 | API 调用记录 |
| GCP Cloud Audit Logs | Admin Activity 默认；Data Access 需手动启用 | — |
| Azure Activity Log | 90 天 | — |

### 5. 云架构设计原则 (Architecting for the Cloud)

#### 迁移策略

| 策略 | 说明 |
|------|------|
| Lift & Shift | 原样迁移，不改代码 |
| Cloud-Native | 利用弹性/敏捷性，失败快/迭代快 |

#### 扩展策略

| 类型 | 方式 | 限制 |
|------|------|------|
| 垂直扩展 (Vertical Scaling) | 换更大的实例 | 有上限，单点故障 |
| 水平扩展 (Horizontal Scaling) | 增加更多实例 | 无理论上限，需无状态 |

#### 无状态 vs 有状态

**无状态 (Stateless)**：服务器不存储会话，任意实例可响应任意请求 → 水平扩展的前提。

**有状态处理方案**：
- 会话亲和性 (Sticky Sessions)：ALB 将同一用户路由到同一实例
- 外部化状态存储：S3 / EFS / DB / Redis (ElastiCache)

#### 基础设施模式

| 概念 | 说明 |
|------|------|
| Pets vs Cattle | 可替换的牲畜 (Cattle)：坏了就换，不修 |
| 配置漂移 (Configuration Drift) | 用不可变基础设施 (Immutable Infrastructure) 解决：替换而非更新 |
| 引导脚本 (Bootstrapping) | User Data / cloud-init，灵活但依赖外部资源 |
| 黄金镜像 (Golden Image / AMI) | 预烘焙，启动快，无外部依赖 |
| 混合模式 (Hybrid) | 镜像 + 启动时少量配置注入 |

#### 松耦合架构

- 服务通过 API / 负载均衡器交互，不硬编码 IP
- 服务发现：ELB 主机名 + Route 53
- 异步集成：在组件之间引入消息队列/流平台

#### 数据库可扩展性

| 机制 | 复制方式 | 用途 |
|------|---------|------|
| 读副本 (Read Replica) | 异步 | 分担读流量；**不可写** |
| Multi-AZ | 同步 | 高可用故障转移；**不分担读** |
| Amazon Aurora | 多主 + 跨区域复制 | 极高读写要求 |

#### 冗余模式

| 模式 | 说明 |
|------|------|
| 热备 (Standby / Active-Passive) | DB 故障转移 |
| 主动-主动 (Active-Active) | LB 分发，所有节点同时处理 |

#### 缓存策略

- 应用层数据缓存：Redis / ElastiCache（内存缓存）
- 减少数据库读压力
- 注意缓存失效策略 (Cache Invalidation)

## 对比表格

### Monolith vs Microservices

| 维度 | 单体 (Monolith) | 微服务 (Microservices) |
|------|----------------|----------------------|
| 部署 | 整体部署 | 独立部署 |
| 扩展 | 全量扩展 | 按需扩展单个服务 |
| 数据库 | 共享 | 每服务独立 |
| 技术栈 | 统一 | 多语言异构 (Polyglot) |
| 开发复杂度 | 低（初期）| 高（需要分布式系统知识）|
| 故障隔离 | 差（一挂全挂）| 好（故障隔离）|
| 适用阶段 | MVP / 小团队 | 规模化 / 大团队 |

### Synchronous vs Asynchronous Communication

| 维度 | 同步 (REST/gRPC) | 异步 (Message Queue) |
|------|----------------|---------------------|
| 耦合 | 时间耦合 | 时间解耦 |
| 响应 | 立即等待 | 非阻塞 |
| 失败处理 | 调用方感知 | DLQ / 重试 |
| 适用 | 需要立即结果 | 后台任务、通知 |

### Lambda vs EC2

| 维度 | AWS Lambda | EC2 |
|------|-----------|-----|
| 服务器管理 | 无需 | 需要 |
| 扩展 | 自动到零 | 需配置 Auto Scaling |
| 计费 | 按请求 + 毫秒 | 按小时/秒（运行即收费）|
| 最长运行 | 15 分钟 | 无限制 |
| 冷启动 | 有 | 无 |
| 适用 | 事件驱动、短时任务 | 长时运行、稳定流量 |

### SNS vs SQS

| 维度 | SNS | SQS |
|------|-----|-----|
| 模式 | Pub/Sub 扇出 | 点对点队列 |
| 推/拉 | 推 | 拉 |
| 消息持久化 | 无 | 最长 14 天 |
| 消费者数量 | 多订阅者各得一份 | 只有一个消费者获得 |
| 典型用例 | 广播通知 | 任务队列、负载均衡 |

## ⭐ 易考点

1. **共享数据库反模式**：微服务最严重的反模式；每个服务必须有自己的数据库；否则破坏独立部署和扩展能力。

2. **SQS 可见性超时 vs DLQ**：消息处理中会被隐藏（默认 30s）而非删除；超过 `maxReceiveCount` 次失败才进 DLQ；需单独设置 DLQ 监控告警。

3. **Lambda 冷启动**：Java/Spring 最慢；减少冷启动：选 Go/Python、减小包体、预置并发 (Provisioned Concurrency)、避免 VPC（或使用 Lambda 的 Hyperplane ENI）。

4. **Security Group vs NACL**：SG 有状态（返回流量自动允许）、仅允许规则；NACL 无状态（返回流量需显式允许）、有优先级顺序、可拒绝。**两者都要配置才能真正限制流量。**

5. **SQL 注入防御 = 参数化查询**：使用托管数据库服务（RDS/Cloud SQL）**不能**防止 SQL 注入；必须在应用层使用 Prepared Statements；这是 OWASP Top 10 中的 Injection 类别。

## 自测题

### 多选题 1

以下哪些陈述关于 AWS Lambda 的描述是**正确的**？（选择所有适用项）

A. Lambda 函数最长可运行 60 分钟  
B. 全局作用域中初始化的数据库连接可以在同一执行环境的多次调用间复用  
C. FIFO SQS 触发 Lambda 可实现精确一次 (Exactly-once) 消息处理  
D. Lambda 函数部署包（ZIP）压缩后最大 50 MB，解压后最大 250 MB  
E. 将 Lambda 放入 VPC 后冷启动时间会增加

<details>
<summary>答案与解析</summary>

**正确答案：B、D、E**

- **A 错误**：Lambda 最长 15 分钟（900 秒），不是 60 分钟（GCP Cloud Functions 第二代才支持 60 分钟）。
- **B 正确**：全局作用域代码在容器初始化时执行一次，连接可在暖启动调用间复用。但不能依赖全局变量保证正确性（容器可能被替换）。
- **C 正确**：FIFO SQS + Lambda 组合可实现精确一次消费保证，避免重复处理。
- **D 正确**：ZIP 部署包 50 MB 压缩 / 250 MB 解压是 Lambda 的硬限制。
- **E 正确**：VPC 内 Lambda 需要创建/复用 ENI（弹性网络接口），增加冷启动延迟约 100-200ms。

</details>

### 多选题 2

关于 AWS SNS 和 SQS 的区别，以下哪些说法**正确**？（选择所有适用项）

A. SQS 是推模型 (Push-based)，SNS 是拉模型 (Pull-based)  
B. SNS 支持将消息同时发送给多个 SQS 队列（Fan-out 模式）  
C. SQS 消息最长可保留 14 天  
D. SNS Standard 主题提供精确一次投递保证  
E. 当需要广播通知给多个下游系统时，SNS 比 SQS 更合适

<details>
<summary>答案与解析</summary>

**正确答案：B、C、E**

- **A 错误**：SNS 是推模型（云主动推送给订阅者），SQS 是拉模型（消费者主动轮询）。题目描述颠倒了。
- **B 正确**：SNS Fan-out 模式：一条消息 → SNS Topic → 多个 SQS 队列同时各收一份，是微服务解耦的常见模式。
- **C 正确**：SQS 消息保留期 1 分钟到 14 天，默认 4 天。SNS 不持久化消息。
- **D 错误**：SNS Standard 提供至少一次 (At-least-once) 投递，可能重复。只有 SNS FIFO 才提供精确一次。
- **E 正确**：广播场景（一对多）正是 SNS 的核心用途；SQS 是点对点（一条消息只被一个消费者处理）。

</details>

### 判断题 1

**陈述**：Security Group（安全组）和 NACL（网络访问控制列表）都是有状态的防火墙，因此只需配置入站规则，出站流量会自动被允许。

**True / False？**

<details>
<summary>答案与解析</summary>

**False（错误）**

- **Security Group 有状态 (Stateful)**：如果允许入站连接，响应流量自动被允许，无需单独配置出站规则。
- **NACL 无状态 (Stateless)**：不跟踪连接状态，入站和出站规则**必须分别配置**。例如允许 HTTP 80 入站，还需要明确允许临时端口 (1024-65535) 的出站，否则响应包无法返回。
- **结论**：两者都配置才能正确控制流量；仅配置安全组不足以保护子网边界，NACL 是子网级别的防线。

</details>

### 判断题 2

**陈述**：在微服务架构中，Saga 模式的核心价值是利用两阶段提交 (Two-Phase Commit, 2PC) 实现跨多个微服务的强一致性 ACID 事务。

**True / False？**

<details>
<summary>答案与解析</summary>

**False（错误）**

- Saga 模式**不使用** 2PC，2PC 在分布式微服务环境中性能差、易造成阻塞。
- Saga 的核心是将分布式事务拆分为**本地事务序列**，通过**补偿事务 (Compensating Transactions)** 在某步失败时回滚已完成的步骤。
- Saga 实现的是**最终一致性 (Eventual Consistency)**，而非强一致性 ACID。
- 两种 Saga 实现方式：
  - **编排型 (Choreography)**：各服务监听事件并触发下一步
  - **协调型 (Orchestration)**：中央协调器指挥各服务

</details>

### 场景题

**场景**：Emma 是一家在线教育平台的架构师，该平台每天有数百万学生上课。系统目前遭遇以下问题：

1. 学生完成作业后，系统需要发送邮件通知、更新学习记录、触发 AI 批改三个操作；目前三个操作串行执行，导致学生等待超过 10 秒。
2. 期末考试季流量峰值是平时的 50 倍，当前固定规格 EC2 集群要么性能不足，要么长期过度配置浪费成本。
3. 安全审计发现数据库连接凭证硬编码在应用的 `config.yaml` 文件中，且已提交到 GitHub。

**问题**：针对以上三个问题，请分别给出 AWS 架构建议，并说明技术原因。

<details>
<summary>答案与解析</summary>

**问题 1：串行操作导致响应慢**

**方案**：引入 **SNS + SQS Fan-out 异步解耦**

- 学生提交作业 → 应用同步写入数据库后立即返回成功（<100ms）
- 发布一条事件到 SNS Topic（作业完成）
- SNS Fan-out 到三个独立 SQS 队列，分别触发：邮件服务 Lambda、学习记录服务、AI 批改服务
- 三个操作**并行异步**执行，学生无需等待

**技术原因**：将同步串行依赖改为异步解耦，消除时间耦合；SNS→多SQS 的 Fan-out 模式天然支持一对多广播，且各服务独立扩展不互相影响。

---

**问题 2：流量峰值 50x，弹性需求**

**方案**：**Auto Scaling Group (ASG) + Application Load Balancer (ALB)** 或 **AWS Lambda**

- 若业务逻辑适合事件驱动且在 15 分钟内完成 → 迁移到 **Lambda**（自动扩缩到零，按毫秒计费，50x 峰值自动处理）
- 若需要长连接或 WebSocket → 使用 **ASG 水平扩展**：设置 CPU/网络指标触发扩容策略，非峰值期缩容到最小实例数
- 考虑在 ALB 前加 **CloudFront CDN** 缓存静态内容，降低源站压力

**技术原因**：垂直扩展有上限；水平扩展（无状态服务器 + LB）可线性扩展；Lambda 的弹性最强，按实际使用付费消除空闲成本。

---

**问题 3：凭证硬编码在 GitHub**

**方案**：立即采取以下步骤

1. **立即吊销/轮换**所有泄露的数据库凭证（GitHub 扫描机器人几分钟内即可发现）
2. 将凭证迁移到 **AWS Secrets Manager**，每次启动时通过 SDK 动态获取，并启用自动轮换
3. 给应用的 EC2/Lambda **IAM Role** 授予最小权限访问 Secrets Manager 的权限，不使用长期访问密钥
4. 在 GitHub 仓库启用 **Secret Scanning**，并在 CI/CD 中加入凭证扫描工具（如 Trufflehog）
5. 审查 `config.yaml` 是否加入 `.gitignore`，将配置文件中的密钥引用改为环境变量（从 Secrets Manager 注入）

**技术原因**：硬编码凭证是 OWASP Cryptographic Failures / Security Misconfiguration 的典型案例。最小权限 + 动态密钥获取 + 自动轮换 = 纵深防御 (Defense in Depth)。

</details>
